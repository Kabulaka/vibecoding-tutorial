# 第四章压缩修订设计

## 背景

当前第四章 `/home/nika/workspace/doc/vibecoding/.worktrees/fourth-chapter-case-study/vibecoding-tutorial/case-study/build-a-real-project.md` 已形成完整主线，但明显超出项目规范中“单文件 800-1500 字、阅读时间 5-10 分钟”的范围。现阶段目标不是重写章节方向，而是在保留案例主线与教程价值的前提下，做一次定向压缩和收束。

## 修订目标

把第四章压缩成一篇更紧凑、推进感更强的案例章节，同时保留以下核心链路：

1. 模糊需求收敛成可执行目标
2. 最小 PRD 与任务拆解
3. AI 产出第一版
4. 人类纠偏与控范围
5. Debug 闭环
6. Review 与交付收口

修订后应继续符合项目文风：简体中文、亲切、实战、略带幽默，但减少重复铺垫与过长类比。

## 推荐方案

采用“强收束版”为主、“局部吸收半结构化版”为辅的压缩策略。

### 主体策略

- 保留现有章节顺序，不改结构骨架。
- 每个步骤尽量压成“1 段核心解释 + 1 组要点/示例”。
- 删除重复表达同一层意思的铺垫句、总结句和比喻句。
- 把抽象说明尽量落回案例动作，减少反复解释“为什么这很重要”。

### 局部增强策略

在适合的位置保留少量列表化表达，帮助读者快速扫描：

- 最小 PRD 定义
- Debug 反馈格式
- Review 检查维度
- 实践部分的操作模板

这样既能保住教程的顺滑叙事，又能提高信息密度。

## 保留重点

修订时必须保留以下信息，不可压掉：

- 案例主题是“本地 Web 版 Prompt / Task 工作台”
- 不引入数据库、登录、权限、多角色、部署等超出范围内容
- 与第三章方法论的承接，尤其是 `../practical-skills/how-to-spec-tasks.md`
- “不是靠神奇 Prompt，而是靠闭环组织项目”的核心判断
- 人和 AI 的角色分工：AI 负责加速，人负责目标、边界、判断和验收

## 主要删减策略

### 1. 删重复铺垫

以下类型优先压缩：

- 同一节中前后反复解释“这不是某个技巧，而是一条完整工作流”
- 连续两段都在说明“不要一开始做太大”
- 一组意思已经通过案例说清，后面又用抽象总结再讲一遍

### 2. 收比喻和幽默

保留少量点到为止的幽默，但避免一个小节里出现多次类比，尤其是会打断推进感的比喻句。

### 3. 压长段解释

第二步到第五步目前都有较长铺陈，修订后统一收成：

- 先讲这一步在案例里的作用
- 再给一个具体示例或要点列表
- 最后用一句话收束

### 4. 收实践与小结

`## 实践` 和 `## 小结` 保留，但都应明显缩短。实践部分保留可直接操作的模板；小结部分只回收主线，不再展开新一轮解释。

## 小修方向

### `### 这次不再讲原则，直接做一个完整项目`

目标：从“章节为什么这样写”快速落到“这次要做什么案例、为什么这个范围刚好”。

处理：压成 2-3 段，减少对前三章学习状态的反复描述。

### `### 第一步：先把模糊想法收敛成可执行目标`

目标：保留“最小 PRD + 任务拆解”的关键教学点。

处理：
- 合并重复解释
- 保留一组清晰的 PRD 最小定义
- 保留一段面向 AI 的示例指令
- 用更短的过渡句承接到下一步

### `### 第二步` 到 `### 第五步`

目标：每一步都讲清动作，但不拖长。

处理：每节控制在“短说明 + 要点/例子”的紧凑格式，减少长段评述。

### `## 实践`

目标：保留可迁移练习模板。

处理：保留五段练习法，但减少额外解释。

### `## 小结`

目标：只收回闭环主线，并自然过渡到第五章。

处理：压缩为 2-3 段，不重复整章内容。

## 修订边界

这次是压缩和收束，不是重写方向。以下内容不在本轮范围内：

- 更换案例主题
- 调整章节大纲
- 新增额外方法论章节
- 扩展产品功能设定
- 引入新工具或新工作流分支

## 完成标准

本轮修订完成后，应满足：

1. 章节仍完整保留“收敛 → 拆解 → 第一版 → 纠偏 → Debug → Review → 收口”主线。
2. 读者能快速看懂每一步在案例中的作用，而不是被重复解释拖慢。
3. 全文明显短于当前版本，朝项目规范的目标区间收敛。
4. 文风依旧亲切实战，但推进更快、更利落。
5. 不引入超出当前任务范围的新内容。
