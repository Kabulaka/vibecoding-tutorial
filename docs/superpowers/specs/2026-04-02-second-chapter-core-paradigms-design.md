# 第二章《核心范式》设计文档

- 日期：2026-04-02
- 范围：`vibecoding-tutorial/core-concepts/` 下第二章的 5 篇内容设计
- 状态：待用户审阅

## 1. 目标

第二章《核心范式》的目标不是继续教读者“怎么更会写 Prompt”，而是推动读者完成一次工作方式升级：从亲自下场逐行实现的开发者，转变为能够定义目标、提供上下文、设置边界、校验结果的“总指挥”。

本章应帮助读者达成 5 个结果：

1. 理解 VibeCoding 中最重要的变化不是工具更强，而是人机分工被重写
2. 理解上下文质量如何直接决定 AI 输出质量
3. 掌握一个可操作的 AI 放权判断框架，知道何时可放手、何时必须审查
4. 理解测试和验收标准为什么是 VibeCoding 的必需品，而不是附属品
5. 建立对 Skills、Hooks、MCP 的整体认知，知道如何把单次协作升级成系统化能力

## 2. 第二章的总设计

第二章采用“总指挥主线驱动”结构。五篇内容不是并列罗列五个知识点，而是沿着“总指挥的五项基本功”递进展开：

1. **先转变角色**：用 `mindset-shift.md` 完成身份升级，建立本章总前提
2. **再补给上下文**：用 `context-is-all.md` 解释 AI 为什么不是玄学，而是吃上下文工作的系统
3. **后划定边界**：用 `ai-trust-model.md` 说明放权的尺度与人工审查的必要性
4. **再加安全带**：用 `test-driven-vibe.md` 解释测试与验收标准如何成为协作护栏
5. **最后系统化增强**：用 `skills-hooks-and-mcp.md` 收束全章，把方法升级为流程能力

整章的主线是：**角色转变 → 上下文供给 → 放权边界 → 验证护栏 → 系统化增强**。

## 3. 五篇文章的职责分工

### 3.1 `mindset-shift.md`

职责：作为第二章开篇，先完成认知立论，回答“为什么 VibeCoding 要求开发者从执行者转向总指挥”。

建议结构：

1. 开场破题：把 AI 当高级补全是低配用法
2. 旧角色的问题：亲自写每一行代码时，AI 只能充当零散助手
3. 新角色的职责：开发者负责目标、约束、验收、决策，AI 负责草拟、搜索、执行、整理
4. 误区提醒：总指挥不是甩手掌柜，人类判断并没有退出现场
5. 小结：后四篇就是总指挥必须掌握的四类能力

写作重点：

- 重点写“分工变化”，不是写工具宣传
- 要让读者感到这是工作方式升级，不是流行词包装
- 结尾自然挂到后四篇，建立第二章地图

### 3.2 `context-is-all.md`

职责：回答“总指挥靠什么指挥”，说明上下文是 AI 产出质量的决定性变量。

建议结构：

1. 开场：为什么一句“帮我改一下这个 bug”经常把 AI 送进沟里
2. 上下文构成：文件、对话、工具结果、限制条件
3. 窗口限制：AI 不是无限记忆体，上下文会拥挤、丢失、稀释
4. 优化策略：如何给足关键材料，而不是把所有材料一股脑塞进去
5. 小结：总指挥的第一项硬功夫不是催 AI，而是喂对上下文

写作重点：

- 要把“上下文管理”讲成实操方法，而非抽象术语
- 可引用第一章中已出现的 Claude Code 使用场景，强调“提供信息”比“润色措辞”更重要
- 这一篇会成为后续 `ai-trust-model.md` 和 `test-driven-vibe.md` 的前置基础

### 3.3 `ai-trust-model.md`

职责：回答“总指挥该放权到什么程度”，给出一套现实可用的 AI 信任判断框架。

建议结构：

1. 开场：真正危险的不是 AI 出错，而是人对风险没有分层
2. 判断维度：可逆性、影响范围、是否可验证
3. 低风险任务：可让 AI 高度自动推进的场景
4. 中高风险任务：必须人工复核、分步授权或先做计划的场景
5. 小结：信任 AI 不是凭感觉，而是按任务风险分层放权

写作重点：

- 不讨论抽象伦理，专注开发现场的授权尺度
- 要让读者形成“先看风险，再看效率”的判断习惯
- 可自然回扣 `context-is-all.md`：上下文越充分，可控性通常越高，但不等于无需审查

### 3.4 `test-driven-vibe.md`

职责：回答“总指挥如何防止翻车”，说明测试和验收标准为什么是 VibeCoding 的安全带。

建议结构：

1. 开场：没有测试的 VibeCoding，本质上是在高速公路上松开方向盘
2. 为什么测试是必需品：AI 生成速度快，放大了验证缺口
3. 如何写可验证需求：把模糊愿望转成验收条件、边界条件、失败标准
4. 测试作为护栏：先定义成功，再让 AI 实现
5. 小结：总指挥不靠直觉确认完成，而靠可验证标准确认完成

写作重点：

- 重点不是系统讲测试理论，而是讲测试在 AI 协作中的控制作用
- 要强调“先定义可验证结果，再让 AI 冲”这条主线
- 应回扣 `ai-trust-model.md`：可验证性越强，可放权程度通常越高

### 3.5 `skills-hooks-and-mcp.md`

职责：作为第二章收束篇，回答“总指挥如何从单次协作升级为系统调度”。

建议结构：

1. 开场：只靠临场聊天，协作质量会高度依赖状态和记忆
2. Skills：把高频工作套路固化为能力模板，明确推荐安装 [superpowers](https://github.com/obra/superpowers)
3. Hooks：在关键事件自动插入检查、提醒或动作，强调它是把协作规则自动执行化的关键一环
4. MCP：把外部工具、知识源与服务接到 AI 工作流中，明确推荐安装 chrome-devtools-mcp、context7、memory、sequential-thinking、spec-workflow、tavily（当 Claude Code 使用非 Claude 模型时，可作为内置 WebSearch 的替代）、[context-mode](https://github.com/mksglu/context-mode)
5. 推荐清单如何协同：说明这些能力分别解决“浏览器调试、文档检索、长期记忆、分步推理、规格驱动、联网搜索、上下文治理”等问题，组合后才能形成稳定工作流
6. 小结：当方法被固化进流程，总指挥开始管理系统，而不是单次对话

写作重点：

- 三者都讲“在工作流中的作用”，推荐清单只说明适用场景与价值，不展开成安装或 API 参考手册
- 要强调它们是协同关系，而不是三个孤立功能点
- 本篇应承担第二章收束任务，把前四篇能力串成一个完整模型
- 对推荐工具的口径使用“建议安装”，但避免写成逐步安装教程

## 4. 示例策略与连续小案例

本章采用“混合型”示例策略：先给命令 / Prompt 片段，再嵌入连续小案例解释其意义。

### 4.1 命令 / Prompt 片段的作用

每篇都应给出 1-2 个短小、可复用的 Claude Code 指令片段，目标是让读者立刻知道“这种情况下该怎么说”。

要求：

- 片段要短，突出结构，不追求工具细节齐全
- 重点展示“总指挥式表达”与“普通提问式表达”的差别
- 片段后应立即解释：为什么这种表达更有效、给 AI 传递了什么关键信息

### 4.2 连续小案例的作用

第二章五篇共用一个轻量连续案例：**给一个已有小项目补一项新功能，并确保不破坏原有行为。**

案例设计原则：

- 足够通用，多数开发者无需行业背景即可理解
- 足够轻量，不会压过“核心范式”本身
- 能自然串起目标定义、上下文补充、风险判断、测试兜底与流程增强

### 4.3 五篇中的案例落点

- `mindset-shift.md`：展示“帮我写”与“目标 + 限制 + 验收”的指挥式提问对比
- `context-is-all.md`：展示上下文不足和上下文充分时，AI 输出质量的差异
- `ai-trust-model.md`：展示同一案例中哪些动作可自动推进，哪些动作必须人工审查
- `test-driven-vibe.md`：展示如何先写验收条件，再让 AI 实现功能
- `skills-hooks-and-mcp.md`：展示当类似任务重复出现时，如何把一次性协作升级成可复用流程

## 5. 第二章的节奏与过渡

第二章的读者体验路径应为：**先完成角色升级，再获得指挥方法，最后看到系统化增强方向。**

### 从第一篇到第二篇

`mindset-shift.md` 结尾应自然引出：

> 如果你已经接受自己不是逐行实现者，而是协作的总指挥，那么下一个问题就是：你到底靠什么来指挥 AI？答案不是语气词，而是上下文。

### 从第二篇到第三篇

`context-is-all.md` 结尾应自然引出：

> 即使你已经学会给足上下文，也不代表所有任务都能放心交给 AI。真正成熟的协作，还需要一套放权边界。

### 从第三篇到第四篇

`ai-trust-model.md` 结尾应自然引出：

> 你可以按风险决定让 AI 跑多远，但如果想让放权更稳，不能只靠经验判断，还需要可验证的安全带。

### 从第四篇到第五篇

`test-driven-vibe.md` 结尾应自然引出：

> 当你已经学会定义目标、管理上下文、控制风险、设置护栏，下一步就不是更努力地下指令，而是把这些方法固化进系统。

### 第二章结尾应传达的状态

读者在读完第二章后，应明确感受到：

- 自己在 VibeCoding 中的核心职责已经发生变化
- AI 输出质量不是玄学，而是可以通过上下文和流程管理来稳定提升
- 信任 AI 不是“敢不敢”，而是“怎么按风险分层放权”
- 测试不是附属工作，而是高速协作的前提
- Skills、Hooks、MCP 是把个人技巧升级为系统能力的关键接口

## 6. 写作边界

第二章需要刻意避免以下问题：

1. **把范式写成口号**：不能只讲“你要当总指挥”，必须给出可执行的方法
2. **把工具写成手册**：Skills、Hooks、MCP 不展开为安装说明或 API 详解
3. **把测试写成独立课程**：聚焦其在 AI 协作中的验证功能，而非覆盖完整测试体系
4. **让案例压过范式**：连续案例只作为承载物，不能喧宾夺主
5. **重复第一章内容**：第二章是在认知入门之上做工作方式升级，不再重复工具初体验叙事

## 7. 成功标准

如果第二章设计成功，读者在本章结束时应达到以下状态：

1. 能用自己的话解释：为什么 VibeCoding 中开发者更像总指挥而不是纯执行者
2. 能识别高质量上下文由哪些部分组成，并知道如何避免无效塞料
3. 能用一个简单框架判断哪些任务可让 AI 自动推进，哪些任务必须人工审查
4. 能先写出可验证需求或测试标准，再让 AI 动手实现
5. 能理解 Skills、Hooks、MCP 分别解决什么问题，以及三者如何协同增强工作流

## 8. 本设计的选择理由

本章采用“总指挥主线 + 混合型示例 + 连续小案例”的方案，而不是“概念分讲”或“重案例驱动”，原因是：

- 用户已明确要求第二章**严格按提纲结构逐篇落地**，因此设计必须服务于既定五篇主题，而不是重排结构
- 用户已明确希望采用**平衡型**写法，因此每篇都需要先讲清概念，再立刻接实操场景
- 用户已明确希望采用**混合型**示例，因此需要同时提供可复用片段与贯穿案例
- 用户已明确希望“角色转变：从开发者到总指挥”成为**强主线**，因此五篇必须围绕这一转变建立递进关系

这个方案最符合当前教程定位：面向有开发经验、但未接触 VibeCoding 的技术人员，帮助他们在第二章中完成从“知道 AI 能帮忙”到“知道如何稳定指挥 AI 协作开发”的升级。
