# 第四章《完整案例》精简设计文档

- 日期：2026-04-02
- 范围：`vibecoding-tutorial/case-study/build-a-real-project.md`
- 状态：待用户审阅

## 1. 目标

本次精简的目标，不是把第四章砍成提纲，也不是把案例写薄，而是在保持“完整案例”教学价值的前提下，把正文收敛得更紧、更顺、更像一条可执行链路。

精简后的第四章仍然围绕**本地 Web 版 Prompt / Task 工作台**展开，继续承担“把前面几章的方法拉到一个真实小项目里跑通”的职责，但表达上要更克制：减少铺垫和重复解释，把篇幅优先留给真正影响协作质量的主线内容。

本次精简后，章节要保住 4 个核心结果：

1. 让读者看到一个模糊想法如何被收敛成可执行目标
2. 让读者看到目标如何继续压成最小 PRD / spec
3. 让读者看到 spec 如何拆成 tasks，并形成实现护栏
4. 让读者看到最小可用版本、验收闭环与 review 如何接上前面的定义工作

## 2. 推荐保留的主线

第四章精简后，正文应严格围绕以下主线推进：

**目标收敛 → 最小 PRD / spec → tasks 拆解 → 最小可用版本 → 验收 / review 闭环**

这条主线之所以必须保留，是因为它刚好对应 VibeCoding 里最关键的一次角色转换：开发者不再只是“让 AI 直接开始写”，而是先定义目标、再压缩范围、再安排步骤，最后用验收与审查把交付兜住。

如果精简后只剩“让 AI 先做一版看看”，那第四章就会退回成一次普通生成演示，失去它作为案例章的价值。

## 3. 建议压缩策略

### 3.1 应优先保留的内容

优先保留以下三类信息：

1. **输入 / 输出 / 约束 / 验收标准**
   - 这是“模糊想法变成工程语言”的关键一步
   - 也是后续最小 PRD 和 task 拆解的依据

2. **spec → tasks 的承接过程**
   - 要明确写出：目标收敛之后，不是立刻开写，而是先形成最小 PRD / spec，再拆成可执行任务
   - 这是当前稿件里最容易被标题跳跃掩盖，但最该保住的部分

3. **最小可验证版本与验收闭环**
   - 要让读者理解“先跑通主流程”不是偷懒，而是因为前面已经把边界和顺序钉住
   - 验收和 review 不是事后补救，而是前面定义工作的自然延伸

### 3.2 可压缩的内容

以下内容可以适度删弱，以换取主线清晰度：

1. 重复强调“这次不做数据库 / 登录 / 权限 / 多角色 / 部署”的铺垫句
2. 解释同一个观点的连续比喻和修辞段
3. 与主线重复表达的“AI 会跑偏 / 人要握方向盘”类段落
4. 过长的情绪性过渡句，只保留最必要的一句即可

### 3.3 不应删掉的内容

以下内容即使精简，也不应删掉：

1. 案例题目必须继续是“本地 Web 版 Prompt / Task 工作台”
2. 必须保留对第三章 `../practical-skills/how-to-spec-tasks.md` 的自然承接
3. 必须保留“输入 / 输出 / 约束 / 验收标准”这一组明确术语
4. 必须保留“先收敛，再实现，再验收”的步骤感

## 4. 当前稿件的主要问题

基于现有文稿，自审后主要发现 3 个问题：

### 4.1 标题顺序不够直白

当前第一步已经在正文里补入“最小 PRD / 任务拆解”的意思，但第二步标题仍偏向“让 AI 先做出可运行的第一版”。这会让读者产生轻微跳跃：刚理解完 spec / tasks 的必要性，章节标题却直接跳到了实现。

### 4.2 主线虽已出现，但承接不够显式

现稿里已经出现了“最小 PRD”“任务拆解”“先跑通主流程”的内容，但它们更多藏在正文解释里，没有被章节标题和结构明确接住。对案例章来说，这会削弱“完整工作流”的可见度。

### 4.3 存在潜在坏链

文内仍引用 `../practical-skills/how-to-spec-tasks.md`，但当前 worktree 未见对应文件。这个问题不影响本次压缩设计本身，但应作为交付前检查项保留。

## 5. 推荐的小修方向

本次精简不建议重写整章，而建议做结构型小修：

1. **把第二步标题改成更靠近 spec 的表达**
   - 例如：`第二步：先用最小 PRD / spec 把实现护栏钉住`

2. **把第三步标题改成更靠近实现交付的表达**
   - 例如：`第三步：让 AI 交付最小可验证版本`

3. **保持后续顺序清晰**
   - 第四步继续写 Debug / 修正闭环
   - 第五步继续写 review / 验收闭环

调整后的顺序建议为：

1. 第一步：先把模糊想法收敛成可执行目标
2. 第二步：先用最小 PRD / spec 把实现护栏钉住
3. 第三步：让 AI 交付最小可验证版本
4. 第四步：当问题出现，进入 Debug 闭环
5. 第五步：在 review 和验收里真正完成交付

## 6. 成功标准

如果这次精简设计是成功的，用户在读第四章时应该获得以下体验：

1. 能清楚看到完整案例的步骤链，而不是只看到一篇泛泛的项目叙事
2. 能明确理解“spec / tasks 并不是额外流程，而是为了让实现更稳”
3. 能感受到篇幅更紧，但信息密度更高，主线更清楚
4. 不会误以为第四章的重点是“让 AI 直接生成第一版”
5. 仍然能自然衔接第三章方法和后续的实现 / 验收闭环

## 7. 本设计的推荐结论

推荐采用**结构小修 + 文案压缩**方案，而不是整章重写。

原因是：当前稿件的问题主要集中在标题承接和部分铺垫偏长，并不存在根本性方向错误。通过调整第二步、第三步的标题与过渡，再删弱少量重复铺垫，就足以把第四章压缩成一条更清晰的“目标 → spec → tasks → MVP → 验收”主线。

这个方案改动小、收益高，也最符合当前“精简而不是重做”的目标。
