# 《使用 Claude Code 构建一个带 AI 总结功能的 Web 笔记工具》二合一设计方案

## 1. 目标与定位

这份方案是一份 **二合一设计稿**，但主次明确：

- **主目标**：服务一场以 Claude Code 为核心的培训演示
- **次目标**：把这场演示自然沉淀为教程第四章《完整案例》的正文

本方案不把重点放在 Web 笔记工具本身，而是把它当成一个足够小、足够真实、足够适合现场展示的载体。真正要展示的是 Claude Code 的操作流：如何给任务、如何观察执行、何时验证、何时介入纠偏，以及如何把等待时间转化为方法论讲解和现场互动的窗口。

因此，这份方案既要满足培训场景下的“可演性”，又要满足教程场景下的“可落稿性”。前者决定节奏与互动，后者决定结构与落稿方式。

## 2. 受众与成功标准

### 2.1 目标受众

受众是有开发经验、但刚接触 VibeCoding 的技术人员。其特点通常包括：

- 能理解 Node.js、TypeScript、Sqlite、Web 项目等基础概念
- 对“AI 写代码”有兴趣，但尚未形成稳定的协作方法
- 容易把 Claude Code 误解为“高级补全工具”或“会聊天的脚手架”

因此，方案不需要花大量篇幅解释基础编程概念，而应重点帮助受众建立一个新的认知：Claude Code 是一个可被指挥、可被观察、可被纠偏的开发代理。

同时，整体风格不宜太板正。讲者话术应带一点幽默，互动节奏要让观众持续参与，而不是坐着看一场无声屏幕录制。

### 2.2 培训成功标准

一场培训演示算成功，不以代码绝对完美为标准，而以以下结果为标准：

1. 观众能看懂 Claude Code 的操作流，而不是只看到“AI 帮忙写代码”
2. 观众能理解人类在协作中承担的角色：给意图、设边界、做验证、做纠偏
3. 观众能意识到等待时间并不是浪费，而是讲述方法论和做互动提问的黄金窗口
4. 即使现场出现错误，也能通过高质量反馈把错误转化为教学素材
5. 观众能跟着演示节奏参与判断，而不是只被动看结果

### 2.3 教程成功标准

作为教程第四章的内容来源，这场演示还需要满足以下要求：

1. 能自然串联第二章的核心范式与第五章的最佳实践
2. 能维持 GitBook 统一的“目标 → 内容 → 实践 → 小结”结构
3. 能让读者看懂完整案例的推进节奏，而不是一堆零散技巧
4. 改写后的正文只需修改 `vibecoding-tutorial/case-study/build-a-real-project.md`，不扩散到其他章节
5. 正文保留一点现场感与轻度幽默，但不写成口语化流水账

## 3. 主线案例定义

### 3.1 案例题目

主线案例固定为：**使用 Claude Code 构建一个带 AI 总结功能的 Web 笔记工具**。

这是一个为了培训与教程双重目标而刻意挑选的题目，不追求“功能大”，追求“流程完整、效果直观、观众一眼能看懂”。

### 3.2 技术主线

主线始终是 Claude Code 在真实完成一个可运行的小项目：

- 初始化一个 Node.js + TypeScript + Sqlite 的 Web 项目
- 实现 add / list / view
- 接入真实 LLM API，实现 summarize
- 运行、验证、局部调试

相比 CLI，Web 项目更直观：观众能直接看到界面、输入框、列表、详情和摘要结果，不需要把注意力分一半去理解命令行输出。

### 3.3 功能范围

案例保留四个核心动作：

- `add`：新增笔记
- `list`：查看笔记列表
- `view`：查看单条笔记详情
- `summarize`：对单条笔记生成 AI 摘要

在 Web 项目里，它们会以页面交互而不是命令形式出现，但教学结构保持不变：

- `add / list / view` 用于建立最小可运行的主流程
- `summarize` 用于把项目从“普通 Web 小工具”升级成“AI 协作产品”

### 3.4 明确边界

本案例明确不扩展以下内容：

- 登录与权限
- 多角色协作
- 部署与发布
- 富文本编辑器
- 标签系统、搜索系统等附加功能
- 复杂状态管理方案
- 与当前主线无关的架构重构

Sqlite 只作为一个足够真实、又不至于把演示拖进数据库泥潭的本地存储方案。

### 3.5 为什么选这个题目

这个题目适合作为二合一案例，原因有五个：

1. **界面反馈更直观**：Web 页面比 CLI 输出更容易让观众一眼看懂
2. **技术栈仍然克制**：Node.js + TypeScript + Sqlite 足够真实，但不会变成大型全栈项目
3. **结构天然分段**：基础功能与 AI summarize 可以自然分成两段
4. **等待节点丰富**：建项目、装依赖、起服务、调 API、修错误都适合穿插讲解
5. **更适合互动**：观众可以更自然地参与判断，比如“这个页面现在算能用了吗？”、“这里如果报错你会先改哪一层？”
## 4. 演示编排总策略

### 4.1 叙事主线

整场演示的主线，不是“教大家做一个 Web 工具”，而是“借一个 Web 工具，演示 Claude Code 的工作方式”。

讲者的角色不是手写实现者，而是操作流讲解者。Claude Code 的角色不是演示背景板，而是现场执行者。

因此整场演示始终围绕四个动作展开：

1. 给任务
2. 看执行
3. 做验证
4. 做纠偏

### 4.2 等待期穿插法

本方案固定采用 **等待期穿插法** 作为核心编排策略。

当 Claude Code 进入以下状态时，讲者切换到方法论讲解：

- 分析任务
- 生成代码
- 安装依赖
- 运行命令
- 修复错误

等待期不空场，而是用于穿插：

- 第二章：核心范式
- 第五章：最佳实践

这样形成的节奏是：

- 你负责给意图
- Claude Code 负责执行
- 你负责解释为什么这样给、为什么现在验证、为什么这样纠偏

### 4.3 设计原则

为了同时兼顾“培训可演”和“教程可写”，演示设计必须遵守以下原则：

- **先讲操作流，再讲工具功能**
- **先讲判断方法，再讲提示词技巧**
- **每个等待点都要能回扣第二章或第五章的概念**
- **每个演示阶段都要能一一映射到第四章正文中的一个小节**

- 今天不是手写 Web 工具
- 今天是看 Claude Code 如何从目标走到结果

这个问题很适合热场，因为大多数人会先怪模型，正好给后面的“总指挥视角”做铺垫。

- Claude Code 不是补全工具，而是可调度的执行代理

### 5.2 0:03 - 0:07 初始化与首轮提示词

#### 演示目标

让 Claude Code 从 0 开始接手一个 Node.js + TypeScript + Sqlite 的 Web 笔记工具项目，但先只做基础页面流程，不做 summarize。

#### 推荐 shell

```bash
mkdir vibe-notes-web
cd vibe-notes-web
claude
```

#### 推荐首轮提示词

```text
帮我从 0 开始创建一个 Node.js + TypeScript + Sqlite 的 Web 笔记工具项目，项目名叫 vibe-notes-web。

需求：
1. 支持新增笔记、列表页、详情页
2. 笔记存储在本地 Sqlite 中
3. 页面交互尽量简单直接
4. 先不要做 summarize
5. 请优先保证主流程可运行，不要过度设计

请你：
- 先规划最小可用实现
- 搭好项目结构
- 实现页面与数据存储
- 安装所需依赖
- 告诉我如何运行验证
```

#### 互动点

可以先问观众一句：

- “如果是你来起手，你会先让 AI 写页面，还是先把数据存储钉住？”

这种问题不难，但很容易把观众从“看热闹”拉进“跟着判断”。

#### 等待期穿插点

当 Claude Code 分析任务和搭项目结构时，讲者应穿插第二章《[上下文管理：如何喂饱 AI 的记忆](../../../vibecoding-tutorial/core-concepts/context-is-all.md)》。

#### 要强调的判断方法

说明为什么第一轮提示词必须把这些信息讲清楚：

- 技术栈
- 功能边界
- 存储方式
- 优先级
- 验收方向

结论应明确为：**上下文不是越多越好，而是越准越好。**

#### 对教程正文的映射

这一段会沉淀为第四章“第一步：先定目标，不急着写代码”。

### 5.3 0:07 - 0:15 基础 CRUD 开发

#### 演示目标

让 Claude Code 完成新增笔记、列表页、详情页的最小可用版本，并通过真实页面验证。

#### 推荐验证命令

```bash
npm install
npm run dev
```

然后在浏览器里依次验证：

1. 打开首页
2. 新增一条笔记
3. 在列表中看到新笔记
4. 点进详情页查看内容

#### 互动点

这一段可以问观众：

- “到这一步你会先验页面长得像不像，还是先验数据真的存进去了？”

然后顺势抛出一句轻松点的话术：

> 页面好看当然加分，但先别让自己被 CSS 哄骗了，数据真落地才算数。

#### 等待期穿插点

这一阶段适合同时穿插：

- 第二章《[什么时候该信 AI，什么时候必须验证](../../../vibecoding-tutorial/core-concepts/ai-trust-model.md)》
- 第五章《[为什么核心不是 Prompt，而是任务设计](../../../vibecoding-tutorial/best-practices/prompt-less-is-more.md)》

#### 要强调的判断方法

这一阶段的核心不是夸赞 AI 生成得快，而是建立 **分层信任**：

- 可以先交给 AI：脚手架、页面骨架、路由、基础 CRUD、Sqlite 读写
- 必须人工验证：依赖是否合理、页面是否真能点通、数据是否真实落库、结果是否符合预期

同时要强调：真正重要的不是 prompt 花活，而是你有没有讲清楚：

- 做什么
- 不做什么
- 怎么算完成

这一段会沉淀为第四章“第二步：让 Claude Code 先把基础 CRUD 跑起来”。

### 5.4 0:15 - 0:23 接入 AI summarize

#### 演示目标

在已跑通的 Web 基础上，为单条笔记接入 `summarize` 按钮，并调用真实 LLM API 生成摘要。

#### 推荐提示词

```text
现在在现有 Web 笔记工具基础上增加 summarize 功能。

目标：
- 用户可以在笔记详情页触发 summarize
- summarize 调用真实的 LLM API 生成摘要
- 尽量复用现有页面风格
- 如果笔记太短，可以直接提示无需总结
- 不要重构已经能工作的新增/列表/详情流程

请你先基于当前代码结构设计最小改动方案，然后直接实现，并告诉我需要什么环境变量以及如何运行。
```

#### 互动点

这里可以现场问一句：

- “如果 AI 现在说要顺手重构一下整个页面结构，你们会让它冲，还是先踩刹车？”

这种互动正好把“最小改动”这个约束讲得更有现场感。

#### 建议补充约束

```text
补充约束：
- 环境变量方式读取 API key
- 摘要输出控制在卡片或详情区域，2 到 4 行即可
- 先以页面可跑通为目标，不要额外加复杂配置系统
```

#### 推荐验证方式

```bash
export OPENAI_API_KEY=your_key_here
npm run dev
```

然后在浏览器里：

1. 打开某条笔记详情页
2. 点击 `Summarize`
3. 观察摘要是否成功显示
4. 检查短笔记是否提示“无需总结”

#### 等待期穿插点

Claude Code 在接 API、调异步逻辑、补环境配置时，讲者应穿插：

- 第二章《[AI 能力扩展三件套](../../../vibecoding-tutorial/core-concepts/skills-hooks-and-mcp.md)》
- 第五章《[成本、限制与不适用场景](../../../vibecoding-tutorial/best-practices/limitations-and-costs.md)》

#### 要强调的判断方法

这一段不是为了炫耀“AI 接 API 很快”，而是为了建立两个认知：

1. Claude Code 可以逐步接入外部能力，今天是 LLM API，往前走还可以是 Skills、Hooks、MCP
2. 真实 API 带来的除了能力增强，还有延迟、成本、配置风险和网络依赖

结论要落在：**AI 很强，但不是零成本、零等待、零出错。**

#### 对教程正文的映射

这一段会沉淀为第四章“第三步：给项目接上 AI summarize”。

### 5.5 0:23 - 0:27 现场纠偏与调试

#### 演示目标

把可能出现的错误转化为高质量反馈示范，展示如何把“跑偏”拉回到正确轨道。

#### 推荐纠偏模板 1：运行失败

```text
`summarize` 功能运行失败了。下面是实际报错：

[把终端报错原样贴进去]

请先定位问题原因，并只做与这个问题直接相关的修改。
约束：
- 不要重构整个项目
- 不要改动新增/列表/详情页的行为
- 修复后请重新运行 summarize 验证
```

#### 推荐纠偏模板 2：输出过于啰嗦

```text
现在 summarize 能运行，但摘要效果太啰嗦。

请调整摘要提示，使输出更像 Web 工具结果：
- 简洁
- 2 到 4 行
- 保留关键信息
- 不要加寒暄语

只修改 summarize 相关逻辑，不要动其他功能。
```

#### 等待期穿插点

这一段适合穿插：

- 第五章《[“及时回退”法则：避免无效迭代](../../../vibecoding-tutorial/best-practices/rewind-not-loop.md)》
- 第五章《[高效协作原则](../../../vibecoding-tutorial/best-practices/dos-and-donts.md)》

#### 要强调的判断方法

讲者要明确示范两件事：

1. 如果 AI 连续几轮都在同一个坑里打转，就不要陪它硬磕；该收缩问题就收缩，该回退就回退
2. 最有效的反馈不是催促，而是高信息量反馈：给现象、给目标、给边界、给范围

#### 互动点

这一段可以直接请观众做判断：

- “如果它连续三轮都在同一个坑里绕，你会继续让它试，还是先回退一步？”

也可以加一句轻松点的话术：

> AI 卡住的时候最怕两件事：一个是它嘴硬，一个是你心软。

这一段会沉淀为第四章“第四步：报错不可怕，关键是怎么纠偏”。

把观众的注意力从“功能是否完成”拉回到“方法是否学会”。

第一，Claude Code 不是补全工具，而是可以被指挥的开发代理。

## 6. 教程落稿映射方案

### 6.1 标题与定位

```md
# 使用 Claude Code 构建一个带 AI 总结功能的 Web 笔记工具
```

标题需要明确案例主题，同时让读者一眼看出这是一个现场演示型案例，而不是抽象理论章节。

### 6.2 固定结构

正文继续保持 GitBook 的统一结构：

- `## 目标`
- `## 内容`
- `## 实践`
- `## 小结`

其中 `## 内容` 下的小节建议固定为：

1. 这次不讲抽象原则，直接跑一遍现场演示
2. 第一步：先定目标，不急着写代码

### 6.3 映射原则

教程正文不是简单转录演示，而是要把演示过程中讲者的判断方法沉淀为读者能复用的文字结构。因此每个演示阶段都需要同时包含：

### 6.5 正文写法要求

正文要满足以下要求：

- 不写成纯演示稿，也不写成纯理论稿
- 保持训练目标清晰：让读者看懂操作流
- 在关键段落保留轻量互动感，让读者能感受到讲者是在带着观众一起判断

## 7. 边界与取舍说明

### 7.1 为什么不做成完整产品

如果把案例扩成完整产品，教学重点会从“操作流”漂移到“功能堆叠”。这样既不利于 30 分钟培训演示，也会让第四章正文失去聚焦。

### 7.2 为什么用真实 API

本方案默认培训环境已准备好可用 API key 或中转环境，因此 `summarize` 采用真实 LLM API，而不是 mock。这一选择不是为了增加炫技感，而是为了让观众看到真实开发中的延迟、成本、环境依赖与验证责任。

### 7.3 为什么不先讲大段理论

本方案不采用“先讲很多理论、再开始演示”的编排，因为那会削弱 Claude Code 现场工作的张力，也不利于把第二章和第五章的知识点绑定到真实等待点上。

### 7.4 为什么不扩写到其他章节

当前设计只服务于第四章案例改写，不扩散到第二章或第五章本身。第二章和第五章在这里承担的是“被引用”和“被串联”的角色，而不是“被重写”的角色。

## 8. 后续实施约束

为了让后续实施计划聚焦且可执行，本方案给出以下边界：

- 先更新设计方案文档

- 后续正文改写阶段只修改 `vibecoding-tutorial/case-study/build-a-real-project.md`
- `vibecoding-tutorial/SUMMARY.md` 与 `vibecoding-tutorial-outline.md` 只作为参考基线，不在本轮改动
- 不在本方案阶段直接改教程正文
