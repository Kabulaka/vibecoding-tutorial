# 第一章《快速开始》设计文档

- 日期：2026-04-01
- 范围：`vibecoding-tutorial/quick-start/` 下第一章的 3 篇内容设计
- 状态：待用户审阅

## 1. 目标

第一章《快速开始》的目标不是把读者一次性训练成熟练的 VibeCoding 使用者，而是让读者在最短路径内完成从陌生到敢开始的转变。

本章应帮助读者达成 3 个结果：

1. 理解 VibeCoding 不是“会写 Prompt”，而是一种 AI 协作开发方式
2. 看懂一次小功能开发的人机协作过程，直观感受到速度优势
3. 拥有最小可用工具链，具备开始尝试的条件

## 2. 第一章的总设计

第一章采用“三步递进”结构：

1. **先理解**：用 `what-is-vibecoding.md` 建立认知框架
2. **再惊艳**：用 `first-vibe.md` 展示一次旁观式的小功能开发协作
3. **后落地**：用 `toolkit-setup.md` 提供最小可用的工具链上手路径

整章的主线是**认知建立**，不是工具堆砌，也不是立即要求读者亲手完成复杂操作。

## 3. 三篇文章的职责分工

### 3.1 `what-is-vibecoding.md`

职责：校正读者认知，回答“VibeCoding 到底是什么、为什么值得学、和传统开发有什么不同”。

建议结构：

1. 开场破题：VibeCoding 不是魔法，也不是万能 Prompt
2. 角色变化：开发者从逐行实现者，转为目标定义者、上下文提供者、结果验收者
3. 为什么值得学：效率提升、试错成本降低、对有开发经验的人尤其友好
4. 边界提醒：AI 会犯错，验证与判断仍然由人负责
5. 小结：本篇建立正确认知，不追求操作细节

写作重点：

- 少讲抽象概念史，多讲开发者工作重心发生了什么变化
- 让读者意识到核心不是“更会写 Prompt”，而是“更会组织任务与验收结果”

### 3.2 `first-vibe.md`

职责：制造第一次数感与速度体感，让读者看到一次完整的人机协作开发闭环。

设计要求：

- 采用**旁观式体验**，而不是要求读者立刻自己跟做
- 选择一个**小而完整的功能开发**作为示例
- 首要强调**速度感**，同时让读者看到人类判断没有消失

建议结构：

1. 任务背景：给出一个简单、常见、能快速完成的功能需求
2. 协作过程展示：需求输入、AI 响应、调整与推进
3. 关键节点点评：哪里体现效率，哪里体现人类的判断与控制
4. 结果验收：如何确认不是“看起来能用”而是真的完成了目标
5. 小结：让读者形成“原来一次 AI 协作开发可以这样跑完”的直觉

案例选择原则：

- **小而完整**：能在短篇幅内讲清楚需求、过程、结果
- **足够常见**：多数开发者无需行业背景就能理解
- **能体现速度优势**：几轮协作就能看到明显产出
- **也能体现人类判断**：至少出现 1-2 次需求澄清、微调或验收动作

适合的例子：

- 给页面增加一个简单表单
- 加一个按钮和交互反馈
- 做一个轻量的接口或前后端联动微功能

不适合的例子：

- 纯静态文案修改
- 太重的业务流程
- 依赖大量前置背景的任务

### 3.3 `toolkit-setup.md`

职责：降低行动门槛，让读者具备最小可用的起步环境。

工具介绍深度采用**轻量上手**策略：不仅知道是什么，还能按最小步骤配起来，但不展开复杂高级配置。

建议结构：

1. 为什么现在就要装工具：因为后续章节会持续使用
2. 核心工具介绍：以 Claude Code 为主，Superpowers / MCP / Hooks 为增强组件
3. 最小安装路径：只保留必要步骤，确保读者能跟上
4. 启用后的第一件事：给一个最小验证动作，确认环境可用
5. 小结：先能起跑，再逐步升级配置

写作重点：

- 强调“够用优先”
- 避免把第一章写成复杂环境部署说明书
- 用最少步骤帮助读者从“理解”走向“开始”

## 4. 三篇之间的节奏与过渡

第一章的读者体验路径应为：**先理解 → 再惊艳 → 后落地**。

### 从第一篇到第二篇

`what-is-vibecoding.md` 结尾应自然引出：

> 概念讲完了，但如果没亲眼看过一次完整的人机协作过程，这些理解仍然容易停留在口号层面。下一篇直接看一次真实而轻量的协作演示。

### 从第二篇到第三篇

`first-vibe.md` 结尾应自然引出：

> 现在你已经看过一次完整的 AI 协作开发闭环，下一步不是继续听概念，而是把最小工具链搭起来，准备自己开始。

### 第一章结尾应传达的状态

读者在读完第一章后，应明确感受到：

- 这不是玄学
- 这不是纯拼 Prompt
- 这套方式确实能提升效率
- 不需要等把所有理论学完才开始使用

## 5. 写作边界

第一章需要刻意避免以下问题：

1. **过度神化 AI**：不能暗示“开发者只要动嘴就够了”
2. **过深原理展开**：上下文管理、可信度模型、Hooks/MCP 深层玩法应留给后续章节
3. **工具配置过载**：只覆盖最小可用路径，不在第一章塞入高级配置清单

## 6. 成功标准

如果第一章设计成功，读者在本章结束时应达到以下状态：

1. 能用自己的话解释 VibeCoding 是一种什么样的协作方式
2. 看懂一次 AI 协作开发的小功能闭环，并感受到速度优势
3. 完成最小工具链准备，具备开始尝试的信心与条件

## 7. 本设计的选择理由

本章采用“认知主线 + 演示支撑 + 工具落地”的方案，而不是“演示先行”或“工具先行”，原因是：

- 用户已明确希望第一章以**认知框架优先**
- 首次体验采用**旁观式**而非跟做式，更适合降低理解门槛
- 示例价值优先强调**速度感**，有助于让读者快速建立兴趣
- 第一章结束目标是让读者**敢开始用**，因此需要用轻量工具上手完成闭环

这个方案最符合当前教程定位：面向有开发经验、但未接触 VibeCoding 的技术人员，在最短路径内完成观念切换与起步准备。
