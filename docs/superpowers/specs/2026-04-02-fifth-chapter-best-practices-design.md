# 第五章：最佳实践 设计文档

## 1. 章节目标与定位

第五章“最佳实践”承担的是整本教程的“防翻车加固”职责，不再引入新的核心能力，而是把前四章已经建立的认知和方法，收束成一套更稳定、更可复用的协作工作法。

它主要回答以下问题：

- 为什么读者明明已经会用 Claude Code，却依然容易越做越乱。
- AI 协作什么时候开始失控，哪些信号意味着结果质量正在退化。
- 如何在效率、质量、成本之间做现实取舍。
- 怎样把“偶尔好用”升级成“长期稳定好用”。

这一章的职责分为三层：

1. **收口**：把前文的零散技巧组织成稳定工作法。
2. **避坑**：集中讲最容易翻车的场景、错误动作和止损方式。
3. **进阶**：给读者一条从“会用”走向“工作流化”的升级路径。

## 2. 目标读者与写法原则

### 2.1 目标读者

- 有开发经验，但刚接触或刚开始稳定使用 VibeCoding 的技术人员。
- 已阅读前四章，对 Claude Code、任务拆解、调试、代码审查已有基本认知。
- 当前最需要的不是“再学一个新技巧”，而是建立一套不容易翻车的长期协作方式。

### 2.2 写法原则

第五章整体采用以下风格：

- **操作手册型**：以规则、判断标准、执行清单为主，而不是抽象方法论长谈。
- **标准篇幅**：每篇控制在 1000-1500 字，兼顾规则、示例、误区与小结。
- **Claude Code 场景优先**：例子尽量围绕写文档、改代码、拆任务、调试等常见协作场景。
- **踩坑避雷指南气质**：重点回答“什么时候会翻车、为什么翻车、如何避免翻车”。

### 2.3 统一结构模板

每篇文档统一采用以下结构：

1. **目标**：这一篇要解决什么问题。
2. **内容**：先讲典型翻车信号，再讲正确原则与判断方式。
3. **实践**：给一个 Claude Code 协作场景，展示错误做法与正确做法。
4. **小结**：给 checklist、判断表或操作规则，帮助读者立即应用。

### 2.4 全章统一要求

- 开头尽快抛出“为什么这里容易翻车”。
- 每篇至少包含一个“错误示例 vs 正确示例”。
- 每篇结尾都给出可执行清单或判断表。
- 不写空泛鸡汤，不堆砌概念定义。
- 不重复展开前文已详细讲过的基础内容，相关概念以链接回指为主。

## 3. 第五章内部逻辑链

第五章 7 篇文档不是并列散文，而是一条连续的防失控逻辑链：

1. **先管住上下文**：`context-limits.md`
2. **再学会止损**：`rewind-not-loop.md`
3. **再把任务拆小**：`epic-to-story.md`
4. **再优化协作动作**：`dos-and-donts.md`
5. **再纠正 Prompt 误解**：`prompt-less-is-more.md`
6. **再讲现实边界**：`limitations-and-costs.md`
7. **最后给进阶出口**：`next-steps.md`

这种顺序的核心价值是：

- 从“为什么开始变差”讲到“怎么避免继续变差”。
- 从“怎么不失控”讲到“怎么长期稳定变强”。
- 前四篇偏实操避坑，第五、六篇负责认知纠偏，第七篇负责升级方向。

## 4. 七篇文档分工说明

### 4.1 `context-limits.md`

**目标**：让读者识别“上下文退化”的早期信号，并知道如何主动控量。

**核心论点**：AI 不是越聊越懂，超过上下文边界后，结果会越来越不稳定。

**建议内容结构**：

- 目标：识别上下文过载。
- 常见信号：开始重复、忽略约束、修一处坏两处、遗漏前文已确认要求。
- 错误做法：继续补更多说明、继续贴更多文件、继续在原线程硬聊。
- 正确做法：缩小处理范围、重开线程、先总结再继续、只保留与当前任务强相关的信息。
- 实践：给一个 Claude Code 场景，展示从“开始跑偏”到“主动收缩上下文”的过程。
- 小结：提供“是否该收缩上下文”的检查表。

**边界约束**：

- 本篇重点是“何时开始变傻”和“怎么控量”。
- 不展开讲“什么时候该回退到上一步”，那是 `rewind-not-loop.md` 的重点。

### 4.2 `rewind-not-loop.md`

**目标**：让读者形成“3 次无效尝试就回退”的止损意识。

**核心论点**：AI 一旦陷入循环，继续催促通常只会把错误放大，而不是自然变好。

**建议内容结构**：

- 目标：识别死循环并及时止损。
- 典型信号：同类修复反复失败、解释越来越自信但结果没有改善、一次次引入新的副作用。
- 错误做法：换个说法继续催、一次性追加更多要求、在错误分支上持续叠加操作。
- 正确做法：回到上一个稳定点，重述问题，明确验收标准，缩小修复范围。
- 实践：给一个修 bug 或改文档的场景，对比“硬磕到底”和“及时 rewind”。
- 小结：给出“什么时候必须回退”的判断规则。

**边界约束**：

- 不重复讲上下文窗口原理。
- 本篇重点是“决策动作”和“止损策略”。

### 4.3 `epic-to-story.md`

**目标**：教读者把大而模糊的目标拆成 AI 可执行的小任务。

**核心论点**：AI 最怕的不是难，而是边界不清、验收不清、依赖关系不清。

**建议内容结构**：

- 目标：把 Epic 变 Story。
- 常见错误：一句话直接丢完整需求，让 AI 同时理解、设计、实现、验证。
- 正确方法：按模块、按流程、按风险点拆解，形成可单独验证的小任务。
- 实践：把一个教程扩写需求或一个功能需求拆成若干 Claude Code 任务。
- 小结：给出“一个好任务的原子性标准”。

**边界约束**：

- 不与第三章 `how-to-spec-tasks.md` 重复讲任务描述格式。
- 本篇重点是“拆解层级”和“控制任务粒度”。

### 4.4 `dos-and-donts.md`

**目标**：沉淀日常协作中的高频有效动作与高频错误动作。

**核心论点**：人与 AI 协作的成败，很多时候不在技术，而在交互习惯。

**建议内容结构**：

- 目标：建立高成功率协作习惯。
- Do：给边界、给验收、及时打断、及时确认、一次只推进一个明确目标。
- Don’t：模糊追问、连续追加新要求、结果刚跑通就盲信、让 AI 在失控状态下继续扩大改动。
- 实践：对比一组“低质量对话”和“高质量对话”。
- 小结：整理成一页协作清单。

**边界约束**：

- 不写成 Prompt 技巧大全。
- 本篇强调“协作动作”和“交互习惯”。

### 4.5 `prompt-less-is-more.md`

**目标**：纠正“学 VibeCoding 就是学写神 Prompt”的误解。

**核心论点**：决定结果质量的，往往不是措辞有多华丽，而是任务定义是否清楚。

**建议内容结构**：

- 目标：拆掉 Prompt 崇拜。
- 常见误区：沉迷收藏模板、追求万能咒语、以为输出不好只是因为措辞不够高级。
- 正确认知：输入、输出、约束、验收标准通常比修辞花样更重要。
- 实践：同一任务下，对比“华丽 Prompt”和“清晰任务说明”的结果差异。
- 小结：把读者注意力从 prompt wording 拉回任务设计。

**边界约束**：

- 不与第三章任务规范完全重合。
- 本篇承担的是“认知纠偏”，而不是“模板教学”。

### 4.6 `limitations-and-costs.md`

**目标**：让读者对成本、延迟、稳定性和适用边界建立诚实预期。

**核心论点**：VibeCoding 很强，但不是零成本、零风险、全场景通用。

**建议内容结构**：

- 目标：建立现实预期。
- 成本：token 成本、等待成本、返工成本、审查成本。
- 限制：上下文有限、外部系统依赖、幻觉风险、验证成本始终存在。
- 不适合场景：高风险生产变更、需求定义不清、缺少验证机制、成本压力敏感的任务。
- 小结：给“这件事该不该先交给 AI”的判断表。

**边界约束**：

- 不能写成唱衰文。
- 重点是“如何理解边界并合理使用”。

### 4.7 `next-steps.md`

**目标**：给读者一个清晰的进阶路线，让教程在“知道了”之后还能继续往下走。

**核心论点**：入门之后，真正的提升来自工作流化，而不是更勤奋地聊天。

**建议内容结构**：

- 目标：从会用走向稳定复用。
- 进阶方向：自定义 Skills、Hooks、MCP、测试、Multi-Agent 协作。
- 推荐顺序：先规范任务，再增加自动化，再扩展外部能力，再考虑多代理协作。
- 实践：给一个“个人 AI 工作流升级路线图”。
- 小结：鼓励读者从一个小自动化开始，而不是一口气搭完整体系。

**边界约束**：

- 不展开讲技术细节，只负责“指路”。
- 重点是建立下一阶段学习地图。

## 5. 跨文档引用策略

第五章需要明确体现“为前文加保险杠”的作用，因此应当主动回链前面章节：

- `context-limits.md` → `../core-concepts/context-is-all.md`
- `rewind-not-loop.md` → 与 `context-limits.md` 形成承接关系
- `epic-to-story.md` → `../practical-skills/how-to-spec-tasks.md`
- `dos-and-donts.md` → 可引用 `rewind-not-loop.md`、`epic-to-story.md`
- `prompt-less-is-more.md` → 回链 `../practical-skills/how-to-spec-tasks.md`
- `limitations-and-costs.md` → 汇总前文关于边界、风险、验证的讨论
- `next-steps.md` → 回链 `../core-concepts/skills-hooks-and-mcp.md`

引用目标不是制造跳转密度，而是帮助读者感受到第五章是在整合前文，而不是另起炉灶。

## 6. 一致性约束与避免重复规则

为保证第五章整体紧凑，每篇都必须遵守以下一致性约束：

1. **每篇只解决一个核心问题**，避免一篇里同时讲上下文、拆任务、成本、进阶。
2. **不重复展开基础概念**，已经在前文讲清楚的概念用一句话提醒并附相对链接即可。
3. **示例必须可落地**，优先采用 Claude Code 中的写文档、改代码、拆任务、调试等场景。
4. **结尾必须产出工具化结果**，如 checklist、判断表、执行规则，而不是只有口号。
5. **风格统一但论点不重复**，所有文档都强调避坑，但每篇要有独立核心命题。

## 7. 建议实施顺序

建议按以下顺序编写第五章，以保证整体逻辑与风格逐步稳定：

1. `context-limits.md`
2. `rewind-not-loop.md`
3. `epic-to-story.md`
4. `dos-and-donts.md`
5. `prompt-less-is-more.md`
6. `limitations-and-costs.md`
7. `next-steps.md`

这个顺序有三个好处：

- 先写最核心的“失控与止损”，便于统一第五章调性。
- 中段再写任务拆解与协作动作，便于形成稳定的方法链。
- 最后写边界与进阶，便于自然收束到教程结束。

## 8. 完成标准

当第五章 7 篇文档全部满足以下条件时，可认为设计被正确落实：

- 每篇都符合“目标 → 内容 → 实践 → 小结”结构。
- 每篇都有清晰的翻车信号、错误做法、正确做法。
- 每篇至少包含一个 Claude Code 场景例子。
- 每篇结尾都有 checklist、判断表或可执行规则。
- 第五章整体读起来像一套连续工作法，而不是 7 篇孤立文章。
- 与前四章形成互补关系，不重复讲基础定义。
