# 第四章《完整案例》设计文档

- 日期：2026-04-02
- 范围：`vibecoding-tutorial/case-study/build-a-real-project.md`
- 状态：待用户审阅

## 1. 目标

第四章《完整案例》的目标，不是继续分散讲解单个原则，而是用一个小而完整的真实案例，把前面三章已经建立的 VibeCoding 认知一次跑通，让读者第一次看到“从模糊想法到可交付结果”的完整闭环。

本章应帮助读者完成 4 个结果：

1. 看到 VibeCoding 在真实开发流程中的完整节奏，而不是零散技巧拼盘
2. 理解“需求收敛 → 任务拆解 → AI 实现 → Debug → Review → 交付”是一个连续过程
3. 体会人类在 AI 协作中的核心职责不是敲代码，而是定义目标、补上下文、控制边界、做出验收判断
4. 带走一套可复用的最小工作流，而不是只记住一个案例故事

## 2. 案例总设计

第四章采用“产品构建型主线”写法，围绕一个 **本地 Web 版 Prompt / Task 工作台** 展开。

这个工具的核心价值很明确：用户输入一段模糊需求，例如“做一个支持标签筛选的任务列表页”，系统将其整理成更清晰的任务说明，并输出结构化任务卡，包括：

- 目标描述
- 输入 / 输出
- 约束条件
- 验收标准
- 可执行子任务

这个题材适合作为完整案例，原因有三点：

1. **足够贴近 VibeCoding 核心主题**：它本身就在解决“如何把模糊需求变成 AI 可执行任务”
2. **足够轻量但完整**：可以覆盖页面、状态、表单、任务拆解与结果展示，又不会引入数据库、鉴权、多角色等分散主线的复杂度
3. **天然承接前三章**：前面讲过的任务设计、上下文管理、测试护栏、Debug 和 Code Review，都能自然在这个案例中出现

这一章不追求做出一个功能很大的产品，而是追求把一条完整工作流跑顺：

**模糊想法 → 可执行目标 → 第一版产物 → 人类纠偏 → 问题定位 → 结果审查 → 最终交付**

## 3. 文章结构设计

第四章建议按以下 7 段推进，形成清晰的过程感。

### 3.1 开场：这次不再讲原则，直接做一个完整项目

职责：告诉读者，这一章的重点不是继续学概念，而是把前面学到的东西串成一次完整协作。

应包含：

- 为什么前三章学完后，需要一个完整案例来建立整体感
- 这次要做的产物是什么：本地 Web 版 Prompt / Task 工作台
- 这次关注的不是“代码多复杂”，而是“流程是否完整”

写作重点：

- 开门见山，快速进入场景
- 让读者明确这是一场“端到端演示”，不是单独技巧展示

### 3.2 第一步：先把模糊想法收敛成可执行目标

职责：展示真正的起点不是开写，而是收敛需求。

应包含：

- 一个故意保持模糊的原始想法，例如“做个能帮我整理需求的小工具”
- 人如何把它补全成明确目标
- 如何把目标整理为输入、输出、约束、验收标准

写作重点：

- 强调这一步是“给 AI 定航线”
- 自然回扣第三章《如何给 AI 下达可执行任务》
- 用简短但清晰的 prompt / 指令示例体现结构化表达

### 3.3 第二步：让 AI 先做出可运行的第一版

职责：展示 VibeCoding 的速度感，但不过度神化 AI。

应包含：

- 通过自然语言描述，让 AI 先生成最小可运行版本
- 第一版包含哪些最核心能力：需求输入区、任务卡展示区、基础交互流程
- 为什么第一版只求“跑通主流程”，不追求一步到位

写作重点：

- 突出“先有可运行版本，再逐轮提高质量”的策略
- 强调 AI 在机械实现和初版搭建上的速度优势
- 不贴大段代码，重点写决策与过程

### 3.4 第三步：第一版出来后，人类开始接管方向盘

职责：说明人类在协作中的核心价值，不是等待，而是判断。

应包含：

- 第一版看起来能用，但哪些地方还需要判断
- 人要检查的内容：是否超范围、交互是否合理、结构是否清晰、文案是否一致
- 为什么“看起来差不多”不等于“可以交付”

写作重点：

- 强化第二章“总指挥”主线
- 让读者感受到：AI 可以提速，人类负责控方向和收口

### 3.5 第四步：当问题出现，进入 Debug 闭环

职责：展示真实协作不会一路顺滑，关键是如何在问题出现时重新组织上下文。

应包含：

- 设置一个真实、轻量的问题，例如：任务卡状态没有及时同步、验收标准展示不完整、输入校验遗漏
- 演示人如何重新补充现象、范围、相关文件和约束，让 AI 聚焦排查
- 展示“别让 AI 盲改，而是先缩小问题空间”的节奏

写作重点：

- 自然回扣第三章《死循环处理：当 AI 陷入反复尝试怎么办》
- 让读者看到 Debug 的关键不是重复试，而是重建上下文与边界

### 3.6 第五步：交付前做一次 AI Code Review

职责：说明交付前的最后一道关，不是“感觉没问题”，而是做一次聚焦型审查。

应包含：

- 审查关注点：过度设计、无关改动、边界遗漏、交互和文案不一致
- 为什么 AI 生成的代码尤其需要看这些问题
- 用轻量 checklist 收束本次案例的验收

写作重点：

- 回扣第三章《审查 AI 的代码》
- 审查不写成正式流程文档，而写成一次真实交付前检查

### 3.7 收尾：交付的不是一段代码，而是一套闭环

职责：总结案例价值，并把读者自然带到第五章《最佳实践》。

应包含：

- 这次案例最终交付了什么
- 为什么它代表的不只是“小工具做完了”，而是一套可复用流程跑通了
- 读者在自己项目里可以迁移的，不是页面细节，而是整套节奏

写作重点：

- 收束到“方法论落地”
- 自然引出后续最佳实践章节

## 4. 文风与内容边界

第四章的文风和边界如下：

### 4.1 文风

- 延续全书风格：简体中文、亲切、实战、略带幽默
- 比前几章更强调“带着读者走完整个过程”的推进感
- 不写成产品说明书，也不写成操作手册流水账

### 4.2 复杂度边界

案例只做一个“小而完整”的本地 Web 工作台，明确不扩展：

- 登录与权限
- 数据库与持久化
- 多角色协作
- 复杂状态管理框架
- 生产级部署细节

原因是：这些内容会显著分散“完整 VibeCoding 闭环演示”这个主目标。

### 4.3 代码呈现边界

- 不贴大段实现代码
- 重点展示关键 prompt、关键决策、关键检查点
- 若需要示例，保持短小，只服务于解释当前步骤

### 4.4 教学边界

- 会引用前三章的关键概念，但不重复大段理论
- 默认读者已经按顺序读过前文
- 重点是“怎么把前文方法串起来”，不是重新讲解这些方法是什么

## 5. 与前三章的承接关系

第四章必须明确承担“串联前三章”的职责。

### 与第一章的承接

- 第一章让读者感受到 AI 能快速产出第一版
- 第四章要把这种“速度感”放入完整流程，而不是停留在初体验层面

### 与第二章的承接

- 第二章强调总指挥角色、上下文质量、放权边界与验证护栏
- 第四章要把这些原则嵌入一个真实案例，让读者看到它们如何同时起作用

### 与第三章的承接

- 第三章给了任务设计、需求拆解、Debug、Code Review、重构协作等单点技巧
- 第四章要把这些能力按真实顺序连起来，形成可迁移的工作流印象

## 6. 成功标准

如果第四章设计成功，读者在读完这一篇后应达到以下状态：

1. 能复述一个完整 VibeCoding 项目从模糊需求到交付的主要阶段
2. 能理解每个阶段中人类和 AI 的职责分工
3. 能意识到“让 AI 写代码”只是其中一步，真正关键的是前后的收敛、纠偏、验证和审查
4. 能把这一套节奏迁移到自己的小项目或真实开发任务中
5. 能自然进入第五章最佳实践，而不会觉得前四章彼此割裂

## 7. 本设计的选择理由

本章采用“开发者工具题材 + 产品构建型主线 + 小而完整的本地 Web 工作台”的方案，原因是：

- 用户已明确选择 **单一主线案例**，因此第四章必须避免分叉案例或多路线写法
- 用户已明确选择 **开发者工具 / 效率工具** 题材，因此案例应贴近开发者自身工作流
- 用户已明确选择 **本地 Web 工作台** 形式，因此案例需要具备页面与交互，以便自然承载完整流程
- 用户已明确选择 **Prompt / Task 工作台** 作为问题场景，因此案例主题应聚焦“把模糊需求整理成可执行任务”这一最贴近 VibeCoding 核心的能力
- 用户已明确选择 **产品构建型主线**，因此整篇文章应以项目推进为骨架，而不是以概念讲解或对话纪实为骨架

这个方案最符合教程当前定位：面向有开发经验、但尚未系统理解 VibeCoding 的技术人员，用一个足够真实、足够小巧、足够完整的案例，帮助他们第一次建立“整套工作流如何闭环”的整体感。
