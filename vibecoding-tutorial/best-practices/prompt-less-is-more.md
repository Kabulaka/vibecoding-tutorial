# 为什么核心不是 Prompt，而是任务设计

## 目标

这一篇要解决一个很常见、也很容易让人越学越偏的误区：很多人以为学 VibeCoding，本质上是在学怎么写出一段惊天地泣鬼神的 Prompt。仿佛只要措辞足够高级、角色设定足够丰满、气势足够恢宏，Claude Code 就会当场开挂。

但很多协作翻车，恰恰就翻在这里：你以为自己是在优化 Prompt，实际上只是把模糊任务包装得更像回事。

真实情况通常更朴素：结果质量往往不取决于你写得像不像“神 Prompt”，而取决于你有没有把任务定义清楚。

如果你还没系统整理过任务说明，可以先回看《[如何给 AI 下达“可执行任务”](../practical-skills/how-to-spec-tasks.md)》。那一篇讲的是任务该怎么组织；这一篇要做的，则是把你的注意力从 wording 拉回任务定义本身。

## 内容

### Prompt 崇拜为什么这么容易出现

“神 Prompt 幻觉”之所以流行，是因为它特别像捷径。

人很容易相信：这次结果不好，不是因为任务模糊、边界不清、验收没说，而是因为自己还没掌握那条传说中的万能咒语。于是开始收藏模板、背角色设定、研究开场白，甚至把一次普通协作写得像在召唤上古贤者。

这件事之所以有吸引力，是因为它让问题看起来很简单：不用重新想任务，只要继续打磨措辞就行。可工程协作不是诗歌比赛。Claude Code 不关心你这段话有没有史诗感，它更关心的是：你到底要我做什么、在哪做、别碰什么、做完怎么算完成。

Prompt 当然不是完全不重要，但它更像表达层，不是任务骨架。骨架没立住，包装越精致，越像把模糊需求认真打扮了一遍。

### 华丽措辞为什么救不了模糊任务

先看一个典型反例：

> 请你以世界级软件架构师、资深产品 strategist 与顶级内容顾问的综合视角，深度优化我们的教程结构与表达，使其更具洞察力、专业度与启发性。

这段话读起来很有气势，但执行价值并不高。因为它没有回答几个最关键的问题：

- 要优化哪一篇，还是整个教程？
- 是改结构、改措辞，还是补案例？
- 能不能调整目录，能不能删内容？
- 输出到底是一份分析，还是直接改文件？
- 什么叫“优化完成”？

这就是很多协作翻车的根源：文字很华丽，任务却很模糊。你让 AI “深度思考”“全面提升”“高级优化”，听起来像是在加 buff，实际上只是给不确定性裹上了一层好看的外壳。

在 Claude Code 场景里，华丽措辞尤其救不了模糊任务。因为它最终还是要落到具体动作上：读哪些文件、改哪个文件、保留哪些约束、交付什么结果。如果这些都没定义，再高级的 wording 也只是提高了误解时的自信度。

### 真正决定结果质量的四件事

真正更值得你反复打磨的，通常只有四件事：**输入具体、输出明确、约束清楚、验收可验证**。

1. **输入具体**：把相关文件、当前现状、已有上下文说清楚，而不是一句“你看下这个问题”。
2. **输出明确**：说清你要的是分析结论、文档草稿、代码修改，还是检查报告。
3. **约束清楚**：哪些文件能动、哪些不能动、是否允许新增依赖、是否只做这一项任务。
4. **验收可验证**：完成后应该出现什么结果，最好具体到包含哪些小节、列出哪些改动、给出哪些验证步骤。

注意，这四件事都不属于 Prompt 修辞学，而属于任务设计。它们决定的是协作结构，不是文案风格。

### 把注意力从 wording 拉回任务定义

下面看一个更贴近 Claude Code 的对比例子。目标都是同一个：写第五章里的《为什么核心不是 Prompt，而是任务设计》。

**错误做法：华丽 Prompt**

> 请你作为一位具有顶尖认知框架、产品洞察与工程哲学的 AI 协作专家，以高度凝练且富有启发性的方式，帮我产出一篇能真正点醒开发者的深度好文。

这段话的问题不是“不礼貌”，而是 Claude Code 很难据此直接执行。它不知道该写到什么范围、必须覆盖什么点、不能写成什么样、做完后又该怎么判断是否达标。

**正确做法：清晰任务说明**

> 只创建 `vibecoding-tutorial/best-practices/prompt-less-is-more.md`。标题为《为什么核心不是 Prompt，而是任务设计》。篇幅约 1000-1500 字，结构固定为“目标 → 内容 → 实践 → 小结”。必须点破“神 Prompt 幻觉”，说明华丽措辞为什么救不了模糊任务，明确写出输入具体、输出明确、约束清楚、验收可验证四件事；必须包含 Claude Code 场景下“华丽 Prompt” vs “清晰任务说明”的对比例子；必须回链 `../practical-skills/how-to-spec-tasks.md`；结尾自然引向 `./limitations-and-costs.md`；不要写成 Prompt 模板教学。

你会发现，第二段并不神秘，甚至有点像项目经理写任务单。但它非常可执行。Claude Code 不需要被你震撼，它需要被你说明白。

所以，真正应该练的，不是如何把一句话修到“像神 Prompt”，而是如何把一个任务定义到“不容易误解”。任务定义对了，普通措辞也能稳定产出；任务定义没立住，华丽措辞只会让偏航显得更体面。

## 实践

拿你最近准备交给 Claude Code 的一个任务，做一次很短的自测：

1. 先写下你原本想发的那句 Prompt。
2. 删掉大部分形容词和角色设定。
3. 补上输入、输出、约束、验收四项。
4. 再问自己一句：如果把这段话发给一个新同事，他能不能直接开始做？你又能不能明确验收？

如果答案是否定的，问题通常不在 wording 不够高级，而在任务定义还不够完整。

## 小结

记住这一页提醒清单：**任务定义优先于措辞。**

- 先定义任务，再润色 wording。
- 先补输入和现状，再要求 AI“深度思考”。
- 先明确输出物，再讨论语气和风格。
- 先说清约束，再期待它不要乱改。
- 先设可验证的验收标准，再评价结果质量。
- 结果不稳时，先检查任务设计，不要立刻掉进“神 Prompt 幻觉”。
- 任何时候，都把注意力拉回任务定义，而不是继续堆修辞。

当你真的把注意力从 Prompt 崇拜拉回任务定义，就会发现协作质量通常会稳定很多。当然，任务定义清楚也不意味着 AI 没有边界、没有成本、没有返工可能。下一篇我们就来正面谈这个现实问题。继续阅读《[成本、限制与不适用场景](./limitations-and-costs.md)》。
