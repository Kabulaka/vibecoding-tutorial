# 成本、限制与不适用场景

## 目标

读完这一篇，你应该能更冷静地判断一件事：AI 协作到底强在哪，也到底卡在哪。这里不是要给 Claude Code 泼冷水，更不是写一篇“别用 AI 了”的唱衰文，而是帮你建立一种更靠谱的工作预期：它确实能提速，但不会免费；它确实能帮你做很多事，但不是所有事都适合先扔给它。

如果你前面已经看过《[上下文的“黄金边界”与防退化指南](./context-limits.md)》和《[为什么核心不是 Prompt，而是任务设计](./prompt-less-is-more.md)》，你会发现这篇其实是在做一次收口：当前文讨论完上下文、风险和任务定义之后，我们终于要正面回答那个很现实的问题——这件事，到底该不该先交给 AI？

## 内容

### 为什么强大不等于零成本

很多人第一次把 Claude Code 用顺了，很容易产生一种错觉：既然它写得快、看得快、还能连着干活，那以后很多任务是不是都可以“低成本外包”给 AI？

先别高兴得太早。AI 协作的优势，主要是把很多原本高摩擦的动作变快了，比如起草、整理、搜索、重构建议、任务拆解和初步实现。但“变快”不等于“没有代价”。它只是把成本结构改写了：你少花了一部分纯手工时间，换来的是新的判断、审查和纠偏成本。

说得更直白一点，Claude Code 不是一个不吃饭、不犯错、不要管理的隐形高级工程师。它更像一个很能打、但需要你持续带方向、看边界、做验收的搭子。你用得越深入，越要接受一个现实：效率提升是真的，管理成本也是真的。

### 你真正要付出的几类成本

很多人只盯着 token 或订阅费用，却忽略了更常见、也更容易把人拖累的四类现实成本。

1. **等待成本**

Claude Code 不会像本地函数调用一样瞬间给结果。你要等它读文件、分析上下文、生成方案、执行命令。有些任务还要来回几轮确认。单次等待不算夸张，但如果你把大量细碎、小到不值得排队的动作都丢给 AI，时间会在一次次等待里被悄悄吃掉。

2. **返工成本**

AI 最擅长的是“先给你一个像样的第一版”，但第一版并不等于最终版。任务边界不清、上下文退化、理解偏差、隐含约束没说透，都会让你进入返工循环。表面上像是 AI 在高速输出，实际上你在高速改回去。前面讲过，AI 一旦开始跑偏，继续硬聊常常不是推进，而是在扩大返工面。

3. **审查成本**

AI 生成得越快，你越容易忘记自己仍然要 review。文档有没有偷换概念，代码有没有绕开边界，命令有没有碰到不该碰的东西，改动有没有引入新副作用，这些都需要你自己确认。它会替你做很多工作，但不会替你承担最终审查责任。

4. **沟通成本**

“让 AI 干活”并不等于“省掉沟通”。很多时候恰恰相反，你要更清楚地描述输入、输出、约束和验收标准。任务定义模糊时，你省下来的不是沟通，而只是把沟通债推迟到了后面，用返工和误解的方式一起还。

错误做法 vs 正确做法，可以先看这个简单对比：

- **错误做法**：觉得 AI 很快，于是把一堆零碎任务、模糊目标、半成品想法一起塞进去，期待它自动整理、自动判断、自动收尾。
- **正确做法**：接受协作本身有成本，只把边界清楚、产出明确、值得并行加速的部分先交给 AI，再把高判断密度的环节留给自己把关。

### AI 协作的天然限制在哪里

成本可以管理，限制更要正视。下面这四条不是“模型下个版本就彻底解决”的小毛病，而是你在使用中必须长期带着的边界意识。

1. **上下文有限**

这件事前面已经详细讲过。Claude Code 很强，但不是无限记忆。线程一长、任务一混、文件一多，它就可能从“理解上下文”退化成“猜你大概想要什么”。所以别把它想成一个能永久无损记住所有前情的队友。

2. **依赖外部信息和工具接入**

很多真实任务不是只靠模型脑补就能完成的。它需要日志、测试结果、数据库状态、接口文档、MCP 工具、权限环境，甚至需要你提供最新业务背景。没有这些，AI 只能在不完整信息上尽力推断。推断有时很有帮助，但它终究不是事实本身。

3. **幻觉不会消失**

这点要说得很诚实：幻觉会降低，但不会消失。它可能引用一个并不存在的函数，误判一个需求前提，或者把“常见做法”说得像“当前项目事实”。最危险的不是它胡说八道，而是它说得很像真的。只要你把它当成永不出错的权威，翻车就只是时间问题。

4. **验证责任不转移**

哪怕命令是 Claude Code 执行的，最后要不要上线、能不能合并、文档能不能发布、操作会不会造成损失，责任都不会自动转移给 AI。它可以帮你写、帮你查、帮你跑，但不能替你承担验证义务。验证责任始终在使用者这边，这条线千万不要因为“它刚才表现很好”就忘掉。

### 哪些场景不该贸然交给 AI 先做

知道边界之后，更关键的是学会判断“不适合先交给 AI”的场景。至少下面这几类，最好先由你自己收口，再决定要不要让 AI 参与。

第一类，**高风险生产变更**。比如线上数据库结构调整、生产配置切换、核心服务重启、流量切换。这类操作不是不能让 AI 辅助分析，而是不该让它直接先动手。因为一旦错了，代价不是“再改回来”这么轻松。

第二类，**权限、支付、删除等高后果操作**。只要涉及权限放开、支付金额、批量删除、账号封禁、数据清理，就要把警觉度拉满。这里最怕的不是小 bug，而是一次错误操作带来不可逆后果。

第三类，**需求未定义清楚**。如果你自己都说不清目标、边界、验收标准，却打算先丢给 AI“帮我想想怎么做”，那它给你的往往不是答案，而是一份看起来很像答案的猜测草稿。需求越糊，AI 越容易把错误方向走得很流畅。

第四类，**没有测试、日志或其他验证手段**。如果你既看不到日志，也没有测试，也没有可检查结果，AI 做出的任何修改都很难被可靠验证。此时把任务先交给 AI，本质上是在盲飞。

再看一个 Claude Code 场景下的对比例子。

**适合先交给 AI：**

你要补一篇 GitBook 文档，目标、结构、必含内容、相对链接都已经明确，还能直接阅读结果做人工 review。这种任务边界清楚、验证便宜、返工成本可控，非常适合先让 Claude Code 起草或改写。

**不适合先交给 AI：**

你准备在生产环境里执行一组涉及权限与删除的数据库操作，但当前只有模糊需求，没有完整变更单，也没有演练环境和回滚验证。这个时候让 Claude Code 先生成并执行操作，不叫提效，叫给事故写前奏。

## 实践

下次你准备把任务交给 Claude Code 前，先做一个 30 秒判断：

- 这件事的目标和边界，我自己是否已经说清楚？
- 出错后代价是“改一改”，还是“出事故”？
- 我有没有测试、日志、预览、diff、review 之类的验证手段？
- 这件事里，AI 该负责起草和加速，还是我其实在逃避本该先做的判断？

如果前两个问题答不清，或者后两个问题的答案明显偏危险，就先别急着交给 AI。先把问题定义清楚、把验证路径准备好，再让它上场，通常更省钱，也更省命。

## 小结

最后给你一张最实用的判断表。以后遇到任务，先别问“AI 能不能做”，先问“这件事该不该先交给 AI”。

| 判断维度 | 适合先交给 AI | 不适合先交给 AI |
| --- | --- | --- |
| 任务定义 | 目标、范围、产出、约束清楚 | 需求模糊，边界说不清 |
| 风险等级 | 出错可回退，代价可控 | 高风险生产变更，错误后果大 |
| 操作类型 | 草稿、分析、重构建议、低风险修改 | 权限、支付、删除等高后果操作 |
| 验证条件 | 有测试、日志、diff、预览、review | 没有测试、日志或其他验证手段 |
| 外部依赖 | 所需信息和工具已接入 | 关键事实缺失，只能靠猜 |
| 人的职责 | 你愿意审查并承担最终验证 | 你想把判断责任一起甩给 AI |

记住这条底线：AI 最适合先接手的是“可描述、可验证、可回退”的工作，不适合先接手的是“高后果、低可见、难验证”的工作。诚实面对边界，不会削弱你和 AI 的协作能力，反而会让你更少翻车、更能长期稳定提效。

当你已经知道哪些坑要避开，下一步就不是“继续更猛地聊”，而是开始思考怎么把有效做法沉淀成自己的工作流。接下来请继续阅读《[下一步，不是更勤奋地聊天，而是工作流升级](./next-steps.md)》。
