# 巨型需求的“原子化”拆解法

## 目标

读完这一篇，你应该能把“做一个完整功能”这种巨型需求，拆成 Claude Code 真能稳定执行、你也能逐步验收的小任务。

如果你已经看过《[如何给 AI 下达“可执行任务”](../practical-skills/how-to-spec-tasks.md)》，那一篇解决的是“怎么把一个任务说清楚”；这一篇要解决的，则是另一个更容易翻车的问题：任务在一开始就太大，根本不适合直接交给 AI。

## 内容

### 为什么一句话丢完整需求最容易失控

很多人和 AI 协作时，最常见、也最容易翻车的幻觉是：**只要我把大需求描述得足够完整，AI 就能一路做完。**

听起来很合理，实际却经常翻车。因为“大需求直接丢给 AI”往往同时包含多个目标、多个文件、多个决策点，AI 很容易一边补细节、一边猜边界，最后产出一个“看起来很努力，实际上很难收尾”的结果。

常见风险有四个：

- 一次改太多，牵连范围失控
- 中途偷偷改了你没授权的部分
- 验收标准模糊，做完也不知道算不算完成
- 回退成本很高，出了问题只能整段重来

错误做法是：

> 帮我把用户系统重构一下，顺便把登录、资料页、权限和后台管理都理顺。

正确做法不是把这句话写得更长，而是先承认一件事：**这不是一个任务，而是一组任务。**

### Epic 和 Story 的差别到底在哪

可以先用两个词把层级想清楚：**Epic** 和 **Story**。

- **Epic**：一个较大的目标集合，通常跨多个模块、多个步骤，不能一次性稳定交付
- **Story**：一个可以独立完成、独立验收的具体工作单元，通常只解决一个明确问题

判断标准别靠感觉，直接看四件事：

1. **是否跨多个子系统**：跨登录、权限、页面、接口，多半还是 Epic
2. **是否需要多次决策**：做着做着要不断选方案，说明颗粒度还太大
3. **是否能一次验收**：如果没法定义一个清晰完成条件，就不是 Story
4. **是否能安全回退**：回滚必须靠祈祷的，说明任务还不够小

一句话记忆：**Epic 是目标包，Story 是交付包。**

比如，“完善后台用户管理”是 Epic；“在用户列表页增加按用户名过滤的前端搜索，并保持现有分页逻辑不变”才更像 Story。

### 怎么按模块、流程、风险点拆任务

拆解大需求时，我建议至少沿三条主轴看：**按模块、按流程、按风险点**。

**1. 按模块拆**

适合结构明确的需求。比如把“订单系统优化”拆成列表页、详情页、状态流转、通知模块。好处是边界清楚，容易控制修改范围。

**2. 按流程拆**

适合用户路径明显的需求。比如先拆“注册 → 登录 → 找回密码 → 首次进入引导”。这样更贴近真实使用过程，也更容易逐步验证。

**3. 按风险点拆**

适合不确定性很高的需求。比如先拆“确认数据库迁移是否安全”“验证旧接口兼容性”“补权限校验测试”。先打掉高风险点，再扩展功能，比一口气铺开更稳。

实际工作里，这三种拆法经常混用。先按模块分大块，再按流程串顺序，最后把最危险的部分单独拎出来。这样你不是在“拆文档”，而是在给 AI 铺一条不容易一脚踩空的施工路线。

### 什么样的任务才算“AI 可执行”

不是所有 Story 都适合直接交给 AI。一个适合 Claude Code 执行的原子任务，至少要满足下面四个标准：

- **目标单一**：只解决一个问题，不顺手夹带别的优化
- **修改范围可控**：最好能预估会动哪些文件、哪些模块
- **验收明确**：完成后看什么结果、跑什么检查，一眼就能判断
- **容易回退**：改坏了可以局部撤销，不会把整条链路一起拖下水

你会发现，这里故意没有重复讲任务说明模板。因为那是《[如何给 AI 下达“可执行任务”](../practical-skills/how-to-spec-tasks.md)》已经解决的问题。本篇更关心的是：**这个任务值不值得被单独发出去。**

一个很实用的判断句是：

> 如果 AI 做完这件事后，我能在 5 分钟内验证结果、在 5 分钟内撤销改动，这个任务通常就足够原子。

## 实践

假设你要用 Claude Code 做一个“大需求”：

> 给现有管理后台补齐用户管理能力，包括搜索、筛选、详情、角色修改、权限校验和操作日志。

如果直接丢给 AI，它大概率会把前端、后端、权限和数据结构一起搅成一锅粥。

更稳的拆法可以是这样：

1. 先确认现有用户列表页相关文件与数据来源，只做分析，不修改代码
2. 在用户列表页增加按用户名搜索，不改后端接口
3. 增加按角色筛选，保留现有分页与排序
4. 新增用户详情抽屉，只展示已有字段，不扩展数据模型
5. 补角色修改入口，但暂时只接已有更新接口
6. 检查角色修改链路里的权限判断，补最小验证步骤
7. 为角色修改操作补后台日志记录
8. 最后统一跑一轮受影响页面和接口的回归检查

这里每一步都比“完善用户管理”小很多，而且都能独立验收。Claude Code 在这种场景下会明显更稳，因为每次只需要盯住一个局部问题，而不是边做边脑补整张蓝图。

错误做法是追求“一次出成品”；正确做法是追求“每次都能安全往前走一小步”。这套拆法同样适用于补小功能、修交互、改文档，或者拆章节任务。

## 小结

判断一个任务够不够原子，可以直接过这份检查表：

### 一个好任务的原子性检查表

- 这件事只对应一个主要目标吗？
- 它的修改范围能提前说清吗？
- 做完后有明确验收结果吗？
- 改坏时能局部回退吗？
- 它不依赖三四个未确认前提才能开始吗？
- Claude Code 执行时，不需要一路替你做产品决策吗？

如果这六条里有两三条答不上来，它大概率还不是 Story，而只是伪装成任务的 Epic。

所以，别急着把巨型需求整段塞给 AI。先拆层级，再控粒度，再逐步推进。这样你得到的不是“偶尔灵光一现的神奇输出”，而是一条可验证、可回退、可持续协作的工作流。下一篇《[高效协作原则](./dos-and-donts.md)》会继续把这些原则收成一组更直接的操作规则。
