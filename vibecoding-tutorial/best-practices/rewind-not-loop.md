# “及时回退”法则：避免无效迭代

## 目标

读完这一篇，你应该能建立一个很实用的止损习惯：当 AI 已经明显跑偏时，不要靠“再试一次”硬扛，而要及时 rewind，回到最后一个稳定点重新组织问题。

如果你刚看完《[上下文的“黄金边界”与防退化指南](./context-limits.md)》，现在可以把那篇的结论再往前推一步：上下文一旦开始失控，下一件要学的事，不是怎么继续解释，而是怎么果断止损。

## 内容

### 为什么继续催，通常不会让局面自动变好

很多人和 AI 协作翻车，不是输在第一次失败，而是输在失败后还想靠“换个说法再来一次”硬磕回来。

问题在于，AI 不是那种被你鼓励两句就能突然开窍的同事。它一旦沿着错误理解走出一段距离，后续每一轮“继续试”，都可能只是在把错误包装得更完整、说得更像那么回事、改得更像在努力。

尤其在 Claude Code 场景里，这个问题会更明显：它能读文件、改文件、跑命令，所以一旦方向错了，产出也会显得非常“勤奋”。你看到它连续修改、连续解释、连续给方案，很容易误以为事情还在推进。实际上，系统也许只是从一个坑，优雅地滑进了另一个坑。

所以这里先给出一个很好用的判断准则：**同类问题连续 3 次无效尝试，就优先回退。**

注意，这里强调的是“同类问题”和“无效尝试”。如果三轮都在修同一个问题，但结果没有实质改善，那就别再把希望寄托在第四次灵光一现上。第四次通常不是奇迹，通常只是更贵的第三次。

### AI 死循环的典型信号

AI 进入死循环时，往往不会大张旗鼓地宣布“我迷路了”。它更常见的表现，是看起来还在认真干活，但局面已经不再变好。

你可以重点观察这些信号：

1. **思路在变，结果没改善**  
   这一轮改参数，下一轮改调用方式，再下一轮换一套解释，但核心报错、核心现象、核心失败点都没动。

2. **解释更自信，但验证仍然失败**  
   它的语气越来越笃定，原因讲得越来越完整，甚至开始引用“应该”“通常”“按理说”，可你一跑验证，还是没过。

3. **不断引入新副作用**  
   原本只是要修一个 bug，结果接口被改了、文案被顺手改了、无关测试也挂了。问题没解决，战线倒是越拉越长。

4. **开始讨论为什么没好，而不是回到哪里**  
   一旦对话重点变成“为什么这个问题这么棘手”“可能是环境比较复杂”“也许再试另一种策略”，而不是“回到哪个稳定点重来”，你们多半已经在错误分支上停留太久了。

这些信号里，最危险的不是某一次失败，而是你开始对失败产生惯性。好像只要 AI 还在输出，局面就还值得继续。其实这时最该做的，往往恰恰是停下来。

### 错误做法：在错误分支上越走越远

最常见的错误做法有两个。

第一种，是**换个说法继续催**。

比如你发现它修了两轮还没修好，就继续说：“再仔细想想”“还是不对，你换个方向”“你再完整检查一下全部相关逻辑”。这种反馈看似在加压，实际是在放宽边界。AI 很可能把你的催促理解成：可以扩大搜索范围、增加改动规模、补更多猜测。

第二种，是**在错误分支上越走越远**。

也就是明明已经怀疑当前改动链条不靠谱了，却舍不得回退，因为“都改到这了”。于是继续叠补丁、补解释、补上下文、补约束，最后变成一个典型场景：你们不是在修最初的问题，而是在维护前几轮错误尝试留下的后果。

这类操作最伤的不是时间，而是判断力。改得越多，你越难分清：现在失败，到底是原问题还没解决，还是后来加出来的新问题。

### 正确做法：回到稳定点重新定义问题

正确做法不是情绪化地推翻重来，而是有步骤地 rewind。

第一步，**回到最后一个稳定点**。  
这个稳定点可以是最后一次确认可运行的 commit、最后一个已验证通过的文件状态，或者至少是“问题还没被扩散前”的版本。

第二步，**重新定义当前唯一目标**。  
不要再说“把这个模块彻底理顺”。改成一句更窄的话，比如：“只修复保存按钮触发 500 的原因，不处理表单重构，不优化提示文案。”

第三步，**明确不能动的部分**。  
把边界讲清楚，例如“不改接口定义”“不改数据库 schema”“不动无关测试”“只允许改这两个文件”。这一步是在防止 AI 又顺手热心过头。

第四步，**先确认验证方法**。  
如果连“怎么证明修好了”都没说清楚，那 AI 很容易用“看起来合理”替代“已经验证”。所以在重新开始前，先约定验证动作：跑哪条命令、看哪个接口、检查什么输出。

一句话总结这个流程：**先回退，再收缩，再设边界，最后才继续。**

## 实践

来看一个 Claude Code 场景。

你让它修一个 bug：用户在设置页点击保存后返回 500。第一轮它改了字段映射；第二轮它怀疑是请求封装，顺手改了 service；第三轮它又说可能是表单状态同步问题，把组件也重构了一遍。结果呢？500 还在，提示文案还坏了，另外两条测试也开始报警。

这就是典型的错误做法：明明已经开始翻车，还想靠继续 loop 把 bug 硬磕出来。看起来每一步都有逻辑，实际上每一步都在扩大不确定性。

更好的做法是直接 rewind，也就是把正确做法落到动作上：

1. 回到最后一次保存功能之外都正常的稳定状态
2. 把目标收缩成“只定位 500 的直接原因”
3. 明确不能动接口和无关 UI
4. 先约定验证方法：点击保存、观察返回体、确认是否仍出现 `email is required`
5. 再让 Claude Code 先分析原因，再给最小修复

你会发现，回退重来并不是认输，而是在把问题重新压回可判断、可验证、可收敛的范围里。

如果你愿意，可以直接套这个提示模板：

```text
先不要继续扩展修改，回到最后一个稳定点后的状态来处理。
当前唯一目标：修复......
不能动的部分：......
验证方法：......
已尝试 3 次且无效：......
请先判断最可能的原因，再给最小改动方案。
```

## 小结

真正成熟的协作习惯，不是让 AI 一直试，而是知道什么时候必须 rewind。

你可以用这条判断规则做结尾检查：**同类问题连续 3 次无效尝试，或已经出现结果没改善、解释更自信但验证失败、持续引入新副作用、开始讨论“为什么还没好”而不是“该回到哪里”这几类信号时，就不要继续 loop，应该立刻回退。**

记住，rewind 不是浪费前面的努力，而是避免后面的努力继续浪费。

当你学会及时回退，下一步就该学会别把任务一下子铺得过大。因为很多“回不来”的局面，往往一开始就是任务尺度太大造成的。继续阅读《[把大任务拆成可执行 Story](./epic-to-story.md)》。
