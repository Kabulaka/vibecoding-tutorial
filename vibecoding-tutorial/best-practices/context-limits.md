# 上下文的“黄金边界”与防退化指南

## 目标

读完这一篇，你应该能判断一件很关键的事：为什么 AI 开局像个清醒的高级同事，聊着聊着却突然开始“犯傻”。更重要的是，你要学会识别上下文退化的早期信号，知道什么时候该收缩、什么时候该重开，而不是把一个线程一路聊成事故现场。

如果你还没读过《[上下文管理：如何喂饱 AI 的记忆](../core-concepts/context-is-all.md)》，建议先看一眼。那一篇讲的是上下文本身是什么；这一篇要讲的，则是另一个更扎心的问题：上下文不是越多越好，喂过量了，AI 也会消化不良。

## 内容

### 为什么 AI 一开始很聪明，后来开始“犯傻”

很多人第一次用 Claude Code 时，都会有一种错觉：前几轮特别顺，照理说越聊它应该越懂我才对。结果现实常常反过来，前面像技术搭子，后面像刚睡醒的实习生。

原因并不神秘。AI 的理解依赖当前还能稳定利用的上下文，而不是“感情越来越深”。当对话、文件、命令输出、限制条件越堆越多时，模型不会自动替你整理成一份完美摘要。它更像是在一张越来越乱的工作台上继续干活：新材料不停堆上来，旧要求被压在下面，最显眼的信息反而不一定最重要。

所以，AI 不是越聊越懂，而是越聊越可能被噪音拖偏。尤其当你把多个目标、多个文件、多个失败尝试塞进同一个线程里时，它就很容易从“理解任务”退化成“尽力猜测你现在最想听什么”。这就是上下文退化。

### 上下文退化有哪些早期信号

真正危险的地方在于：退化通常不是突然死机，而是慢慢变钝。常见的早期信号，至少有这几类：

1. **重复前文**：它开始反复讲你们已经确认过的内容，像在用字数掩盖理解没跟上。
2. **忽略限制**：你明明说过“不要改接口”“只改这一章”“先分析再动手”，它却像没听见一样直接越线。
3. **修一处坏一处**：刚修完 A，B 又坏了；再补 B，C 又冒出来。表面上在推进，实际上是在失控扩散。
4. **回答完整但命中率下降**：措辞依然自信，结构依然完整，甚至看起来更像那么回事，但真正命中的点越来越少。

请注意，最容易误导人的恰恰是第四种。很多人一看回答“挺完整”，就误以为 AI 状态还行。其实这时，它往往已经从“基于上下文推理”滑向“基于经验模板拼答案”了。

### 错误做法：越乱越加料

上下文一退化，人最本能的反应通常不是收缩，而是继续加料：

- 继续补更多说明
- 继续贴更多文件
- 一次性塞更多背景

这三招看起来像是在“帮 AI”，实际上很像给一个已经看花眼的人再发三份需求文档。它不会更清楚，只会更容易把重点埋掉。

最典型的错误推进是这样的：

“你刚才没改对，我再补充一下。这个模块和订单、支付、通知也有关系，顺便参考这些旧实现。另外这个需求其实还要兼容之前那个临时方案，最好一起优化。”

恭喜，你不是在纠偏，你是在把一个局部任务升级成多线程灾难。AI 本来只是上下文开始发胀，你这一脚油门踩下去，直接把它送进了雾区。

### 正确做法：控量、聚焦、重开线程

更靠谱的处理方式，通常就四个字：少而关键。

第一，**控量**。只保留当前问题真正需要的信息。和这一步无关的背景，先别一起端上桌。

第二，**聚焦**。一次只解决一个明确目标。不要一边修 bug，一边重构结构，一边补文档，一边顺手优化命名。Claude Code 再能打，也不该被你安排成四线程搬砖工。

第三，**重开线程**。如果当前对话已经出现明显退化，不要迷信“再解释一次它就懂了”。新开一个线程，把有效信息压缩成干净版本，通常比在旧线程里硬抢救更便宜。

第四，**让 AI 先复述理解**。在真正执行前，先让它用几句话复述：当前目标是什么、限制是什么、准备改哪里。它复述不准，你就立刻知道上下文还不够干净；它复述准了，再推进，成功率会高很多。

## 实践

来看一个 Claude Code 里的对比例子。场景是：你在写第五章文档，本来只想修改一篇最佳实践内容。

### 错误推进

你对 Claude Code 说：

> 把第五章整体顺一遍，顺手把上下文、止损、任务拆解三篇都补齐。前面我提过很多要求，你参考一下之前的讨论，再把相关文件都看看，能优化的地方一起优化。

这类说法的问题非常集中：

- 目标不止一个
- 范围模糊
- 依赖前文零散讨论
- 默认允许扩散修改

结果往往是：它会读很多、说很多、改很多，但你会越来越难判断现在到底在完成哪件事。然后它开始重复前文、忽略限制、修一处坏一处，最后回答看起来很完整，命中率却越来越低。

### 正确推进

更好的说法会像这样：

> 只处理 `best-practices/context-limits.md` 这一篇。
>
> 目标：写清楚上下文为什么会退化，以及如何主动收缩上下文。
> 必含内容：早期信号、错误做法、正确做法、Claude Code 对比例子、结尾检查表。
> 约束：只新增这一个文件；回链 `../core-concepts/context-is-all.md`；结尾自然引向 `./rewind-not-loop.md`。
> 开始前请先复述你的理解，再执行。

这段话的好处不在于“更正式”，而在于它帮 Claude Code 把注意力锁在一个可控范围里。任务单一、边界清晰、验收明确，还多了一步复述确认。你不是在催 AI 快点干活，而是在帮它保持清醒。

## 小结

上下文管理的关键，不只是“喂够”，更是“别喂撑”。一旦出现退化信号，继续往里塞信息，通常不是补救，而是在加速翻车。

下面这份检查表，可以当作你的“是否该收缩上下文”快速判断器：

- [ ] AI 是否开始重复前文，而不是推进新信息？
- [ ] AI 是否忽略了已经明确的限制条件？
- [ ] 最近两三次修改，是否出现修一处坏一处？
- [ ] 回答看起来很完整，但真正命中的点是否在下降？
- [ ] 当前线程里，是否已经混入多个目标、多个文件、多个失败分支？
- [ ] 你是否已经忍不住继续补更多说明、贴更多文件、一次性塞更多背景？
- [ ] 这时候，是否更适合收缩范围、重开线程，并先让 AI 复述理解？

如果上面连续勾中三项以上，别硬聊了。先收缩上下文，通常比继续推进更明智。
