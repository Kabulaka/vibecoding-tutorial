# 进阶：构建自动化 AI 开发工作流

## 目标

读完这一篇，你应该能把一个关键认知钉牢：入门 VibeCoding 之后，真正的提升来自工作流，而不是更勤奋地聊天。

前面几章已经帮你建立了基本功：你知道怎么定义任务、管理上下文、控制风险、用测试兜底，也知道 [Skills、Hooks、MCP](../core-concepts/skills-hooks-and-mcp.md) 这些能力各自是干什么的。现在的问题不再是“我会不会用 Claude Code”，而是“我能不能把偶尔做对，升级成稳定做对”。

这一篇不会展开讲技术细节，而是给你一张下一阶段的学习地图：从单次协作，走向可复用、可检查、可持续优化的 AI 开发工作流。

## 内容

### 为什么真正的提升来自工作流，而不是更勤奋地聊天

很多人会卡在一个很微妙的阶段：已经能把 Claude Code 用起来了，也确实感受到提效，但每次效果都像开盲盒。顺的时候像开挂，不顺的时候像在和一位热情但跑偏的实习生深夜拉扯。

问题通常不在于你不够努力，而在于好结果还主要依赖临场发挥。你这次说清楚了目标、边界和验收，所以结果不错；下次赶时间，少交代一句限制，AI 就可能一路冲到沟里。真正限制上限的，不是你有没有再多聊十轮，而是你有没有把那些“这次之所以做对”的动作，沉淀成默认流程。

错误做法 vs 正确做法，可以先看一个对比：

- **错误做法**：每次都临场组织需求、临场补约束、临场想起要 review，指望自己在忙乱中永远记得所有好习惯。
- **正确做法**：把高频协作动作变成固定套路，把关键检查变成默认动作，让工作质量不再只看当天状态。

所以，下一阶段的重点不是“学会更会聊”，而是“学会搭工作流”。聊天是接口，工作流才是稳定性的来源。

### 从“会用”到“可复用”要跨过哪几步

从基础使用走向工作流化，通常要跨过四步，而且顺序最好别跳。

**第一步，先把任务定义和验收标准写清楚。**  
这是整个升级路线的起点。你要先把“目标是什么、边界在哪、做完怎么算完成”说清楚。没有这一步，后面无论接多少工具，都只是把模糊放大。工作流不是替代思考，而是把已经想清楚的协作方式固化下来。

**第二步，再固化高频流程。**  
当你发现自己总在重复同一种协作节奏，比如“先澄清需求，再拆计划，再执行，再 review”，就说明这部分值得沉淀成固定套路。这里最典型的位置就是 Skills：它负责把高频、稳定、可复用的协作方法变成可直接调用的工作方式，而不是每次从零口述流程。

**第三步，给关键节点加自动检查。**  
流程稳定后，下一步不是继续堆命令，而是找出最容易翻车的几个节点，比如开始动手前是否明确约束、完成之后是否跑验证、改完之后是否检查副作用。这时 Hooks 和测试就开始上场了。Hooks 负责“在关键时刻自动提醒或触发动作”，测试负责“用结果验证这次改动到底稳不稳”。一个管纪律，一个管证据。

**第四步，最后再接外部工具和多代理协作。**  
当你的基础流程已经跑顺，再接 MCP 才真正有价值。因为这时外部工具不是来凑热闹，而是来补能力：查文档、看浏览器现场、联网搜索、记住项目偏好、处理复杂推理。再往后，多代理协作才适合登场，用来并行拆分不同子任务。否则你只是把混乱从一个 AI，升级成一群 AI 一起混乱。

### Skills、Hooks、MCP、测试怎么逐步接起来

你可以把这四类能力放在同一条流水线上理解。

- **Skills**：负责固化流程。它回答的是“这类任务通常应该按什么顺序做”。
- **Hooks**：负责卡关键节点。它回答的是“哪些动作不能靠记性，应该自动提醒或自动触发”。
- **测试**：负责验证结果。它回答的是“这次修改到底是真的完成了，还是只是看起来像完成了”。
- **MCP**：负责扩展外部能力。它回答的是“当 AI 只靠当前对话不够时，去哪里接真实信息和真实工具”。

一个比较稳的接法通常是这样的：先把需求说明和验收标准写清楚，再用 Skills 固定协作顺序；等你知道哪些地方最容易漏，就在这些节点上加 Hooks；然后用测试把“我觉得没问题”变成“我证明它没问题”；最后再按需要接 MCP 和多代理，让工作流能访问文档、浏览器、联网资料、长期记忆，以及并行执行能力。

重点不是把名词背下来，而是知道它们各守哪一段工位。Skills 管流程，Hooks 管纪律，测试管验证，MCP 管外援。位置一清楚，升级就不会乱。

### 不要一口气搭整套系统

这里最容易踩的坑，就是刚学会一点自动化，就想一口气搭完整体系：自定义一堆 Skills、配满 Hooks、接十几个 MCP、再喊来多代理一起开工。气势很足，结果往往是你先把自己维护累了。

更稳的做法永远是小步升级。先解决一个真实痛点，再把那一个动作沉淀下来。比如你总忘记写验收标准，那就先固定任务模板；你总在收尾时忘记跑验证，那就先给完成节点加检查；你总查官方文档查到怀疑人生，那就先接一个文档类 MCP。每次只升级一小段，才能知道这段到底有没有价值。

- **错误做法**：还没形成稳定协作习惯，就急着配置一整套“未来很强”的系统，最后只得到一套需要你天天伺候的半成品。
- **正确做法**：围绕一个真实高频问题做最小升级，跑顺了再加下一层，让工作流像搭积木，而不是像一次性装修全屋智能后天天找遥控器。

## 实践

下面给你一个“个人 AI 工作流升级路线图”示例。它不是标准答案，但很适合作为下一阶段的起步版本。

**阶段 1：能稳定描述任务**
- 写任务时固定包含目标、范围、约束、验收标准。
- 每次只推进一个清晰目标，不让 AI 同时设计、实现、调试、收尾。

**阶段 2：固化自己的高频流程**
- 把常见协作节奏固定下来，比如“先澄清 → 再拆解 → 再执行 → 最后验证”。
- 开始复用适合自己的 Skills，而不是每次从零说明做事顺序。

**阶段 3：给关键节点上护栏**
- 在容易漏的节点加 Hooks，比如动手前提醒看约束、完成后提醒跑验证。
- 给常见任务补上基础测试，让修改结果有证据可查。

**阶段 4：按需接入外部能力**
- 查文档时接文档类 MCP，查当前资料时接联网搜索，调试前端时接浏览器工具，处理复杂分析时接推理工具。
- 只接当前真的能提高成功率的外部能力，不做“工具收藏家”。

**阶段 5：开始尝试多代理协作**
- 先从边界清楚、可并行的小任务开始，比如一个代理查资料，一个代理改文档，一个代理做验证。
- 你自己仍然是总指挥，负责拆任务、定边界、做最终验收，而不是把责任外包给一群分身。

如果你现在刚走到第二章或第三章末尾，看到这里不用焦虑。你不需要明天就变成“AI 工作流架构师”。你只需要问自己一个问题：我最常重复、最常漏、最容易翻车的那一步，能不能先做成默认动作？从这一步开始，升级就已经发生了。

## 小结

到这里，第五章也该收束了，整本教程想交给你的主线，其实已经很明确。

VibeCoding 不是把键盘交给 AI，也不是比赛谁更会写神秘 Prompt，而是学会作为总指挥去定义目标、设置边界、安排流程、加上护栏、验证结果。前面几章帮你建立了基本功，第五章则是在反复提醒你：想长期稳定提效，关键不是更猛地聊，而是把有效做法沉淀成工作流。

所以，读完这本书之后，最值得带走的不是某一条命令、某一个模板，甚至也不是某一个工具名，而是一种升级顺序：先把任务讲清楚，再把流程固化，再给关键节点加检查，最后接入外部能力和多代理协作，并且始终坚持小步升级。

### 工作流升级的收尾检查表

- 先固定一个高频任务的目标、范围、约束和验收。
- 先把常用协作顺序沉淀下来，再考虑自动化。
- 先给最容易翻车的节点加检查，再扩展工具接入。
- 先验证每次升级确实有收益，再决定要不要继续加层。
- 先做一个小而稳的默认流程，不一口气搭完整系统。

如果你能开始这样和 AI 协作，那你就已经不只是“会用 Claude Code 的人”，而是在建立一套真正属于自己的 AI 开发系统。教程到这里结束，但你的工作流升级，才刚刚开始。
