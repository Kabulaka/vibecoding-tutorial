# 审查 AI 的代码

## 目标

读完这一篇，你应该建立一个很务实的观念：AI 生成的代码，不该用“像不像我会写的”来判断，而该用“它有没有真的满足需求、有没有越界、有没有埋坑”来判断。

前面几篇讲的是怎么给任务、怎么拆任务、怎么打断循环；这一篇讲的是协作的最后一道关：怎么把“看起来能用”变成“可以放心收货”。

## 内容

先说结论：**审查 AI 代码，重点不是挑语法毛病，而是查工程风险。**

你可以把检查重点压缩成四个问题：

1. **有没有做对？**
2. **有没有多做？**
3. **有没有乱做？**
4. **有没有漏做？**

### 1. 有没有做对？

先看它是不是完成了原任务。比如你让它“给列表加本地搜索”，它最后是不是按要求过滤指定字段？空关键词是否恢复全量？原来的分页和排序还正常吗？

### 2. 有没有多做？

这是 AI 常见的“热心病”。你让它修一个表单 bug，它顺手把组件结构也重构了；你让它改一个按钮文案，它顺手把样式系统也整理了。表面像帮忙，实际是在扩大风险面。

### 3. 有没有乱做？

这里主要看实现方式是否靠谱。常见问题包括：

- 用了项目里根本不存在的 API 或工具
- 假设了错误的数据结构
- 为了修一个小问题，引入不必要的复杂度
- 复制出一段相似但不统一的逻辑
- 绕开真正问题，只做表面补丁

### 4. 有没有漏做？

AI 很容易完成主体功能，却漏掉边界和收尾，比如失败提示、空状态、验证步骤，或者与原有约束的冲突。

### 短案例

假设你让 AI 做一个搜索框。结果看起来不错：输入框有了，过滤逻辑也有了。但你一细看：

- 它新增了一个只用一次的 helper
- 它顺手拆了原列表组件
- 它过滤的是 `description`，不是你要求的 `title`
- 它清空关键词时没有恢复分页状态

这就是典型的“看起来像完成，实际上没完全对”。

## 实践

下面这份快速清单很适合日常回看：

- **对需求**：它真的实现了我要求的行为吗？
- **对范围**：它有没有改我没授权它改的东西？
- **对复杂度**：这个实现是不是比任务本身复杂太多？
- **对真实性**：它用到的函数、类型、接口在项目里真的存在吗？
- **对边界**：空值、失败、回退这些最基本情况考虑了吗？
- **对验证**：我能不能用一个最小步骤快速确认它说的是真的？

你也可以把它压成一句口头禅：

> 先看是否做对，再看是否做多，再看是否做怪，最后看是否漏边界。

## 小结

AI 代码审查的重点，不是审美，而是风险控制。你不用要求它每次都像手工雕刻的一样优雅，但至少要确认：功能是真的，范围是对的，实现是靠谱的，边界没有被悄悄漏掉。
