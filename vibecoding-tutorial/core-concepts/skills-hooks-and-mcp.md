# AI 能力扩展三件套

## 目标

读完这一篇，你应该能理解一件很关键的事：如果说前面四篇解决的是“怎么把一次 AI 协作做对”，那么这一篇解决的就是“怎么让下次也更容易做对”。

你会知道，为什么只靠临场聊天推进开发会越来越不稳；也会知道为什么应该先把 `superpowers` 这样的主工作流能力用起来，再按场景补上 `chrome-devtools-mcp`、`context7`、`memory`、`sequential-thinking`、`spec-workflow`、`tavily`、`context-mode` 这些扩展能力。它们不是一堆零散插件，而是在不同层面把单次配合升级成可复用流程的关键拼图。

如果上一篇《[自动化测试：Vibe Coding 的安全带](./test-driven-vibe.md)》讲的是“先设护栏，再让 AI 开跑”，这一篇要讲的就是：护栏不能每次都靠你现场手搓，最好把它们直接装进工作流。

## 内容

### 只靠临场聊天，为什么协作会越来越不稳

很多人刚开始用 AI 协作时，感觉很爽：想到什么就说什么，AI 也经常能给出八九不离十的结果。问题在于，这种合作方式很吃状态。

今天你脑子清醒、上下文给得全、验收条件也想明白了，效果就不错；明天你赶时间、少说一句限制、忘了补一个关键文件，AI 就可能开始热情地朝错误方向冲刺。它不是突然变笨了，而是你的协作方式还停留在“临场发挥”。

而临场发挥最大的问题，不是偶尔出错，而是不可复用。你这次之所以做对，可能只是因为你刚好记得：先补背景、再讲边界、最后列验收。可一旦任务变多、节奏变快，这些好习惯很容易漏掉。于是协作质量开始像抽卡：有时很神，有时很玄。

所以第二章真正想把你推到的位置，不只是“更会提问”，而是“开始管理一套稳定的人机协作系统”。前面几篇里你已经知道要定义目标、补上下文、按风险放权、用测试兜底；接下来要做的，就是把这些原则从脑内备忘录，变成默认执行的流程。

### Skills：把好做法固化成可复用套路

**Skills** 可以理解成给 AI 配好的工作套路。你不用每次都从零解释“先帮我想方案，再拆计划，再执行，再验证”，而是可以把这类高频协作方法固化成可重复调用的能力。

它的价值不在于让 prompt 变短，而在于让合作方式更稳定。总指挥最怕的不是多说两句话，而是每次合作步骤都不一致：今天先写代码再补计划，明天先测试后改实现，后天大家又忘了收尾验收。Skill 的作用，就是把“理想合作顺序”固化下来。

这里我会明确建议优先安装 `superpowers`。原因很简单：它不是只给你几个炫技命令，而是在很多关键节点上提醒你用更稳的方式协作，比如先 brainstorm、先写 plan、先做验证、收尾前再检查。更重要的是，在这个项目语境里，它本身就已经覆盖了 `spec / plan / tasks` 这一整套主工作流。

当你开始频繁遇到类似任务时，Skill 的意义会越来越明显。你管理的不是一句 prompt，而是一套反复可用的协作模板。对大多数读者来说，先把 `superpowers` 用顺，往往比一开始并列装很多“看起来也能做流程化”的工具更稳。

### Hooks：把关键检查变成自动触发

如果说 Skills 解决的是“怎么做更稳”，那么 **Hooks** 解决的是“怎么别忘了做”。

很多协作事故，其实不是不会，而是忘了。忘了先看规范，忘了写验收，忘了执行验证，忘了在某个动作前提醒 AI 不要乱改无关文件。你当然可以每次都靠自觉，但自觉这东西在 deadline 面前通常不太耐用。

Hooks 的价值，就是把这些关键规则变成自动触发。比如在某类操作前插入提醒，在执行完成前要求先验证，在读取大输出时提醒改用更省上下文的方式。这样一来，流程纪律就不再只靠记忆维持，而是开始由系统兜底。

对总指挥来说，这非常重要。因为你的职责不是亲自盯住每一步，而是确保关键步骤不会被跳过。Hooks 并不会替你做判断，但它能帮你把“哪些地方必须停一下、看一下、验一下”固定下来。它让流程有了默认护栏，而不是靠人临场想起系安全带。

### MCP：把外部能力接进工作流

**MCP** 的作用，是把 AI 从“只会看当前对话”扩展成“能接工具、知识和服务一起工作”。

这也是为什么我建议安装一组明确的 MCP 工具，因为它们补的是不同能力缺口，而不是互相替代：

- `chrome-devtools-mcp`：建议安装。它负责浏览器调试，适合查页面行为、网络请求、控制台报错这类前端现场问题。你不用对着截图猜半天，能直接让 AI 看到浏览器里的真实情况。
- `context7`：建议安装。它负责文档检索，适合查框架、库、API 的最新官方用法。比起让 AI 凭印象回忆，直接拿文档说话通常稳得多。
- `memory`：建议安装。它负责长期记忆，适合保存项目偏好、团队约定、常见限制。这样很多“我们之前不是说过吗”的内容，就不用每次重讲一遍。
- `sequential-thinking`：按需补充。它适合你已经知道问题很复杂、很绕、很难一下想清楚的时候用。它的价值，不是接管整套流程，而是帮你把一个棘手问题一层层拆开，少一点拍脑袋，多一点按步骤思考。它和 `superpowers` 有一点重叠，但更像卡住时的推理补强。
- `spec-workflow`：按需补充。它适合你已经决定要把需求做得更正式、更可追踪的时候用。比如你不满足于“这次讲清楚了就行”，而是希望把 spec、plan、task 状态持续挂在一条清楚的流程里。这类事情 `superpowers` 本身也能覆盖不少，所以它不是必装；只有当你真的需要更强的规格流转感时，再加上它会更顺手。
- `tavily`：建议安装。它负责联网搜索，适合查当前信息、线上资料和外部参考。在 Claude Code 使用非 Claude 模型时，它也能作为内置 WebSearch 的可靠替代方案。
- `context-mode`：建议安装。它负责上下文治理，适合处理大输出、长日志、多文件摘要这类容易把上下文窗口塞爆的场景。你前面学的“喂对上下文”，在这里终于有了工程化抓手。

这些能力各有分工，合在一起才像一支完整的外援队：有人查资料，有人看现场，有人记偏好，有人做拆解，有人控规格，有人管上下文。

### 推荐清单如何协同

把它们放在一张图里理解，会更清楚：

- Skills 负责固化套路
- Hooks 负责固化纪律
- MCP 负责扩展能力

比如你要推进一个新需求，`superpowers` 可以作为主工作流，提醒你别跳过 brainstorm、plan、verification 这些关键节奏；Hooks 可以在关键动作前后自动插入检查；如果你后面发现这个需求值得长期追踪，再把 `spec-workflow` 接进来，让 spec 和 task 状态挂得更稳；`context7` 与 `tavily` 帮你补外部知识；`memory` 记住团队约定；`sequential-thinking` 则在你碰到复杂判断时，帮你把问题一层层想清楚；`chrome-devtools-mcp` 协助调试实际页面；`context-mode` 则确保这些过程不会把上下文搅成一锅粥。

这时候你会发现，自己管理的已经不是“一次对话怎么说”，而是“一套协作系统怎么跑”。这才是总指挥视角真正成熟的地方。

## 实践

还是回到我们这一整章反复使用的连续案例：给已有用户列表页补一个搜索能力，并确保不破坏原有排序、分页和空状态逻辑。

如果这只是一次性需求，你当然可以临场对 AI 说清楚目标、文件、边界和验收；但如果类似任务会不断出现，比如以后还要补筛选、批量操作、导出、详情联动，那继续每次手写同一套说明，就会越来越像在重复抄作业。

更稳的做法，是把这次协作升级成可复用流程。

第一步，先用 `superpowers` 这类 Skills 固化合作顺序。比如先 brainstorm，再写 plan，再执行，再 verification。这样你不需要每次都重新提醒“别跳步骤”，也能先把 Claude Code 里的主工作流跑顺。

第二步，如果你发现这类需求会反复出现，或者你就是想把过程记得更正式一点，再接入 `spec-workflow`。这样以后碰到类似需求，不再只是“这次讲清楚了”，而是能把目标、范围、验收和任务拆分持续挂在一条更可追踪的流程里。

第三步，用 Hooks 在关键节点自动卡位。比如开始执行前提醒查看约束，完成前提醒跑验证，读取大输出时提醒用更合适的上下文处理方式。很多原本靠你临时想起的纪律，会变成默认动作。

第四步，用 MCP 工具把外部能力接进来：需要查框架写法时用 `context7`，需要查当前资料时用 `tavily`，需要看浏览器现场时用 `chrome-devtools-mcp`，需要处理复杂推理时用 `sequential-thinking`，需要保留项目偏好时用 `memory`，需要控制大体量上下文时用 `context-mode`。

这套变化的重点，不是工具变多了，而是经验开始沉淀了。原来你是“这次刚好做对”；现在你是在搭一条“下次更容易做对”的流水线。一次性协作，升级成了可复用流程。

## 小结

到这里，第二章真正想建立的东西就完整了。

你已经知道，自己在 VibeCoding 里不再只是逐行实现的人，而是负责定义目标、供给上下文、划定边界、设置护栏、调度系统的总指挥。前面几篇解决的是协作原则，这一篇解决的是怎么把原则固化成系统能力。

所以这篇的结论并不是“多装点工具就会变强”。真正的重点是：在需要时优先配好合适的 Skills、Hooks 与 MCP，让好做法可以重复、让关键检查会自动发生、让 AI 能接入更真实的外部能力。这样你管理的就不再只是一次 prompt，而是一套越来越稳的工作流。

如果第二章是“建立范式”，那么下一章就该进入“怎么把范式落地”。第三章会继续把这里讲到的系统化思路，正式落到日常开发里的 spec、任务拆分和执行节奏上。接下来请继续阅读《[如何给 AI 下达“可执行任务”](../practical-skills/how-to-spec-tasks.md)》。
