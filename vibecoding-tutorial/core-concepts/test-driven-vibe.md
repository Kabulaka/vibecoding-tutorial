# 自动化测试：Vibe Coding 的安全带

## 目标

读完这一篇，你应该能把“测试”从一项可有可无的善后工作，重新理解成 Vibe Coding 里的安全带：不是为了显得专业，而是为了把 AI 的输出变成可验证、可交付、可放心继续迭代的结果。

如果上一篇《[AI 信任模型：什么时候相信，什么时候验证](./ai-trust-model.md)》讲的是“不同风险要配不同授权”，那么这一篇讲的就是：当你决定让 AI 多跑几步时，靠什么防止它一路跑进沟里。信任模型决定你放权到哪，测试决定你怎么刹车、怎么验收、怎么确认车还在路上。

## 内容

很多团队一提到 AI 提效，第一反应都是“让它赶紧写”。这很符合人性，但也很符合事故学。

没有测试时，风险会被成倍放大。因为 AI 生成代码的速度远快于人工逐行审查的速度，你今天不是少了一层保护，而是少了一整套减震系统。改一个函数时没测试，问题可能只是一个小 bug；改三个文件、顺手重构两段逻辑、再补一层状态管理时没测试，问题就会从“看起来差不多”升级成“哪里坏了都说不清”。

更麻烦的是，AI 会天然放大验证缺口。人类开发者写完代码，多少还记得自己犹豫过哪几个边界；AI 不会，它会把一份语气很自信的结果递给你，像个交卷速度过快但草稿纸完全没保存的同学。只要你的需求还是模糊的，比如“优化一下体验”“把搜索做好”“修一下提交流程”，它就很容易在错误目标上高速发挥。不是它故意乱来，而是你给它的终点本来就只写了“前方大概有个地方”。

所以在 Vibe Coding 里，真正重要的不是“先写代码”，而是“先定义成功”。对总指挥来说，测试不是开发末尾补上的仪式感，而是你手里最硬的一根控制杆：它负责把“我想要什么”翻译成“什么算完成、什么算跑偏、什么必须拦下”。测试的本质，不是先选 Jest、Playwright 还是别的工具；测试的本质，是把模糊需求翻译成可验证结果。

一个简单判断标准是：如果一句需求无法回答“怎么证明它已经完成”，那它就还不是一个适合交给 AI 直接实现的指令。

比如：

- “把搜索做一下”太模糊。
- “输入关键字后，列表应只显示名称或邮箱中包含关键字的用户；无匹配时显示空状态；清空输入后恢复完整列表”就可验证多了。

前者是愿望，后者才是验收条件。前者适合开会时说，后者才适合让 AI 动手。

这也是为什么要先写验收、后做实现。你先把成功长什么样定义清楚，AI 才是在朝目标施工；否则它只是在根据你的语气猜测老板心情。说得直白一点：没有测试，AI 写的是“也许能用”；有了测试，AI 才有机会写成“已经被证明能用”。

## 实践

我们继续沿用前面的连续案例：给用户列表页补一个搜索能力。

如果直接对 AI 说“给用户列表加搜索，并把体验做顺一点”，它大概率会很积极：加输入框、加过滤、顺手改状态、也许再补个 loading。看起来忙忙碌碌，但你很难确认它到底改对了什么。

更稳的做法是先写验收闭环，再让它实现。

### 第一步：先定义验收结果

先把需求改写成可验证的结果：

1. 初始进入页面时，显示全部用户。
2. 输入 `li` 时，名称或邮箱包含 `li` 的用户会被保留，其余隐藏。
3. 没有匹配结果时，页面显示“未找到匹配用户”。
4. 清空搜索词后，列表恢复完整数据。
5. 已有排序与分页行为不被破坏。

你会发现，这时需求已经从“做个搜索”变成了“什么算做成”。这一步其实就是把上一篇里说的“验证责任”落到了纸面上：不是先相信 AI 会懂，而是先把判断标准说清楚。

### 第二步：让 AI 先补验收，再做实现

这时给 AI 的 prompt 就可以非常具体：

```text
请为用户列表页的搜索功能按“先验收、后实现”的方式推进。

验收标准：
1. 初始显示全部用户。
2. 输入 `li` 时，仅显示名称或邮箱包含 `li` 的用户。
3. 无匹配时显示“未找到匹配用户”。
4. 清空输入后恢复完整列表。
5. 不破坏现有排序与分页行为。

请按下面顺序执行：
- 先把上述验收标准转换为自动化测试或测试清单
- 明确指出当前实现若不满足，会在哪些断言上失败
- 再根据这些验收结果修改实现
- 最后列出你改了哪些文件，以及我该如何最小验证

约束：
- 先不要扩展新功能
- 不要改无关文件
- 如果排序或分页行为不明确，先提问，不要猜
```

这个 prompt 的关键，不在于“写得像不像咒语”，而在于它先定义了成功，再授权 AI 去实现。这样即使 AI 第一次没写对，你也知道它错在第几条验收上，而不是只能盯着页面发呆，嘴里念叨“我感觉哪里不太对”。

### 第三步：形成闭环

真正的闭环不是“AI 改完了”，而是：

- 验收先定义
- 测试先暴露缺口
- 实现再去补缺口
- 最后用测试确认结果

这才是 Vibe Coding 里最稳的节奏。你不是让 AI 自由发挥，而是在给它一条带护栏的跑道。

## 小结

自动化测试在 Vibe Coding 里不是额外负担，而是把速度变成可控速度的那根安全带。没有测试，AI 会把小风险放大成连锁问题；需求越模糊，这个放大器越吓人。先写验收、后做实现，本质上是在把“我想要什么”翻译成“我怎么证明它已经被做好了”。

所以，别再把测试理解成开发结束后的收尾动作。对 AI 协作来说，它更像施工前拉好的警戒线、导航里的终点坐标、以及上一篇《[AI 信任模型：什么时候相信，什么时候验证](./ai-trust-model.md)》里那套验证责任的具体落点。换句话说，测试就是总指挥用来控节奏、卡边界、做验收的那套硬规则：你不是靠感觉确认“应该差不多了”，而是靠它判断“哪里能放手，哪里必须停下”。

当你学会先定义成功，再让 AI 实现，你就不只是写得更快，而是更稳。下一篇，我们继续把这套“先设护栏，再放手执行”的方法往前推进，看看如何借助 Skills、Hooks 和 MCP，把这些规则进一步固化进工作流。请继续阅读《[AI 能力扩展三件套](./skills-hooks-and-mcp.md)》。
