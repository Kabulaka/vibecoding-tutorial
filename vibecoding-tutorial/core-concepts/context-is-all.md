# 上下文管理：如何喂饱 AI 的记忆

## 目标

读完这一篇，你应该能回答三个很关键的问题：AI 到底靠什么理解你的任务？为什么它有时前一轮还很聪明，后一轮突然像失忆？以及在日常协作里，怎样用更少的话，喂给它更有用的上下文？

如果说上一篇《[角色转变：从“开发者”到“总指挥”](./commander-vs-coder.md)》讲的是身份切换，那么这一篇讲的就是总指挥的补给学。你不是只会下命令，还得知道该给部队发地图、口粮和坐标。不给也能冲，但大概率会冲到沟里。

## 内容

先说结论：AI 的表现，通常不是由“它今天聪不聪明”决定，而是由你给它的上下文质量决定。

这里的上下文，不只是你眼前那一句 Prompt。对 Claude Code 这类工具来说，上下文通常由四部分构成：

1. 文件内容：相关源码、配置、文档、接口定义、报错日志。
2. 对话历史：你前面说过什么、改过什么、否定过什么、确认过什么。
3. 工具结果：搜索结果、测试输出、git diff、诊断信息、命令执行反馈。
4. 约束条件：不能改哪些文件、要兼容什么版本、不要新增依赖、风格遵循什么规范。

把它想成 AI 的工作台会比较容易理解。文件是原材料，对话是任务背景，工具结果是现场勘查，约束是施工红线。四样都齐了，它就更像一个靠谱同事；缺两样以上，它就容易进入“我先猜一下”的危险模式。

很多人第一次用 AI 写代码时，会误以为“只要会写 Prompt 就够了”。其实 Prompt 只是开场白，上下文才是主菜。你说一句“修一下这个 bug”，如果没附报错、没给相关文件、没说明预期行为，那它能做的往往不是修 bug，而是参与一场高风险猜谜游戏。

接着要理解第二个现实：上下文窗口不是无限的。

模型能处理的信息量是有上限的，这就是所谓的 context window。超过窗口，不是说 AI 会礼貌地提醒你“抱歉，我吃不下了”，而更常见的情况是：旧信息被稀释、关键约束被忽略、它开始抓住最近几条话当圣旨。于是你明明十轮之前强调过“不要改接口”，它第十一轮还是把 API 改了，还一脸无辜。

这不是它故意叛逆，而是窗口限制带来的自然后果。上下文一多，模型会优先关注更近、更新、更显眼的信息；那些藏在前面对话里的关键要求，如果没有被重新整理和强化，就很容易掉队。

所以，上下文管理的核心，不是“把所有东西一股脑塞进去”，而是“把当前任务真正需要的东西，按优先级喂进去”。常见优化策略有这几类：

第一，给相关，不给全部。不要一上来就把整个仓库、半个月聊天记录和三屏截图一起扔进去。当前只改登录页，就优先给登录组件、鉴权逻辑、接口约定和报错信息；无关模块先别请进来围观。

第二，给结构，不给散装。与其零零碎碎补充十次信息，不如一开始就按“目标、现状、限制、验收”组织好。AI 最怕的不是信息多，而是信息像抽屉里打翻的螺丝钉，哪儿都是，就是不好捡。

第三，给事实，不给脑补。比如“这里有问题”远不如“点击保存后返回 500，后端日志提示 user_id 不能为空”有用；“参考之前的实现”远不如“参考 `auth/login.ts` 里的错误处理方式”有用。

第四，长对话要定期压缩。聊到后面时，主动做一次中场总结很重要。你可以明确重申：当前目标是什么、已经完成了什么、还剩什么、哪些约束依然有效。这样相当于帮 AI 手动做一次记忆整理。

第五，约束要反复出现。那些不能碰的边界，不要只在第一轮说一次。像“不要新增依赖”“只改这个目录”“先分析再改”这类规则，应该在关键轮次重复出现，因为它们往往比实现细节更容易在长对话里丢失。

说得再直白一点：好上下文不是越多越好，而是越准越好。它像给厨师备菜，不是把冰箱整个搬过去，而是把今晚要做这道菜需要的食材和要求摆好。

## 实践

下面看一个非常典型的对比。场景：你希望 AI 帮你修复一个表单提交失败的问题。

### 上下文不足

```bash
claude
```

```text
帮我修一下表单提交失败的问题。
```

这类说法的问题非常集中：

- 没说是哪个表单
- 没说失败现象是什么
- 没说相关文件在哪里
- 没说限制条件
- 没说什么算修好

AI 接到这种任务，通常会先猜你说的是哪个页面，再猜你说的“失败”是前端校验、接口 500，还是数据格式错。猜对了是运气，猜错了你们俩一起加班。

### 上下文充分

```bash
claude
```

```text
请帮我分析并修复用户设置页的表单提交流程。

目标：点击“保存”后应成功更新昵称与邮箱，并显示成功提示。
现象：当前点击保存后前端提示 Request failed with status 500。
相关上下文：
- 表单组件在 src/pages/settings.tsx
- 提交逻辑在 src/services/profile.ts
- 后端接口约定见 docs/profile-api.md
- 最新报错里返回 message: "email is required"
约束：
- 不要修改后端接口
- 不要新增依赖
- 先分析原因，再给出修改
- 修改完成后列出改动文件，并给出最小验证步骤
```

你会发现，后者并没有神秘到哪里去，只是把 AI 真正需要的信息一次性交代清楚了。

这个版本里，文件、对话目标、工具结果、约束条件四类上下文都齐了：

- 文件：组件、服务、接口文档位置明确
- 对话：目标和当前现象说清楚了
- 工具结果：500 与具体报错信息给到了
- 约束：不改后端、不加依赖、先分析后修改

这就是“喂饱 AI 的记忆”的基本方法。不是让它背更多，而是让它更快抓住重点。

如果任务更大，比如跨多个模块重构，就更要分批投喂。先让它理解局部，再让它汇总状态，再推进下一步。别把一个大项目一次性塞进去，指望它像神仙一样永不走神。它是模型，不是许愿池。

## 小结

上下文管理，是 VibeCoding 里最容易被低估、但最影响结果稳定性的能力。AI 不是凭空理解任务，它依赖你提供的文件、对话、工具结果和约束来拼出“现在到底要做什么”。

真正成熟的协作方式，不是拼命写更花的 Prompt，而是像上一篇《[角色转变：从“开发者”到“总指挥”](./commander-vs-coder.md)》说的那样，学会组织战场信息、控制注意力分配、及时重申边界。你给得越准，它偏得越少；你给得越乱，它就越像在雾里开车。
