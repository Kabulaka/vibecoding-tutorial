# 从 0 到交付：端到端实战演示

## 目标

读完这一篇，你应该第一次真正看到：VibeCoding 不是几招零散技巧拼起来的热闹场面，而是一条能从模糊想法一路推进到可交付结果的完整工作流。

前面三章，你已经分别学过角色切换、Context 管理、任务设计、Debug 和 Code Review。这一章不再单独讲哪一招更厉害，而是把这些动作按真实顺序串起来，跑一遍完整闭环。你会看到，人和 AI 不是谁替代谁，而是一个负责定航线，一个负责加速度。

## 内容

### 这次不再讲原则，直接做一个完整项目

第四章不再拆开讲技巧，而是把前面学过的动作按真实节奏串起来，走一遍完整协作。

这次我们故意把范围收小，只做一个本地 Web 小工具：**Prompt / Task 工作台**。它的任务很单纯：把一句模糊需求先收敛，再整理成更适合和 AI 协作的任务卡。重点不是把产品做大，而是把流程走完整。

所以这章先不扩展到数据库、登录、权限、多角色协作或部署。不是它们不重要，而是这次先把“一个人带着 AI 稳稳交付一个小项目”讲明白。只要这个工作台能把需求收敛、任务展示和主流程跑通，它就已经完成教学任务。

### 第一步：先把模糊想法收敛成可执行目标

很多项目一上来就想对 AI 说“来，写吧”。更稳的起点其实是先把目标钉住。

假设你脑子里只有一句话：

> 我想做个小工具，帮我把模糊需求整理一下，最好还能顺手拆成任务。

这句话很真实，也很不够。它没有讲清输入是什么、输出长什么样、这次明确不做什么、做到什么程度算完成。直接开工，AI 多半会一边写一边补设定，最后你再一边看一边把它拽回来。

所以第一步先别急着写代码，而是先把模糊需求收敛成一个最小 PRD，把目标、边界、输出和验收钉住。在这个案例里，最小 PRD 可以先写成这样：

- **项目**：本地 Web 版 Prompt / Task 工作台
- **输入**：一段自然语言需求描述
- **输出**：一组结构化任务卡
- **卡片内容**：目标、输入 / 输出、约束条件、验收标准、可执行子任务
- **形式**：本地 Web 单页应用，先跑通一次输入到整理结果的主流程
- **这次不做**：数据库、登录、权限、持久化，以及其他额外扩展
- **验收标准**：输入一段模糊需求后，页面能产出清晰、可阅读、可继续加工的任务卡

到这里，项目就从“有个点子”变成了“可以执行的目标”。但还差一步：把最小 PRD 再翻译成 tasks 拆解。因为 tasks 不是装饰，它是实现护栏，负责把“先做什么、后做什么、哪些先别碰”说清楚。这样 AI 拿到的就不是一团抽象愿望，而是一条有边界、有顺序的执行路径。

如果你对这一步还不够熟，可以回看前一章《[如何给 AI 下达“可执行任务”](../practical-skills/how-to-spec-tasks.md)》。这里我们只是把同样的方法放进真实项目里：先收敛成最小 PRD，再翻译成 tasks，然后再让 AI 开工。
### 第二步：让 AI 先做出可运行的第一版

这一步最容易跑偏的地方，是还没开工就想把第二版、第三版甚至未来架构一起想完。更稳的做法是：**先让 AI 交一个最小可验证版本。**

注意，不是“随便写一版看看”。在 AI 开工前，人已经先用最小 PRD 钉住目标、边界、输出和验收，又用 tasks 排好主流程顺序。所以第一版不是粗糙试水，而是带着护栏去验证最核心链路：输入一段需求，点击整理，页面能产出结构化任务卡。

给 AI 的要求也要收紧：先做输入区、触发按钮、结果展示区和最基本的处理逻辑；样式优化、附加功能、额外抽象先别碰。先验证三件事：能不能输入，能不能生成，生成结果像不像一组真能继续执行的任务卡。只要这条链路成立，后面才有东西可迭代。

### 第三步：第一版出来后，人类开始接管方向盘

第一版一出来，人就该把方向盘握回来。你的任务主要有三件。

第一，**控范围**：守住这次只做当前闭环，别让 AI 顺手把教学案例长成小产品。第二，**判质量**：检查页面结构、任务卡字段和交互是否真的有用，而不是“看起来像完成了”。第三，**压发挥**：反馈要具体，告诉它保留什么、删掉什么、改短什么、补强什么，别给它即兴表演空间。

一句话总结：AI 负责加速，人类负责定边界、做判断、持续校正。

### 第四步：当问题出现，进入 Debug 闭环

真实项目里，第一版能跑不代表问题消失。比如这个案例里，长一点的输入可能会让“约束条件”和“验收标准”混在一起。这时别只说“结果不太对”，要把 Debug 信息补成一个闭环。

- **现象**：哪里错了，表现是什么
- **范围**：只在哪些输入或场景下出现
- **约束**：这轮不要改什么，只聚焦最相关的一层
- **期望结果**：修完以后，应该稳定看到什么效果

比如你可以要求：长需求输入时字段别再混在一起；短输入基本正常；先别重做页面和加功能；目标是无论输入长短，任务卡都能稳定区分两个字段。这样 AI 才知道该修哪里、别碰哪里、修到什么算完成。

这也是前面《[让 AI 陷入 Debug 循环怎么办](../practical-skills/loop-and-debug.md)》讲的同一套方法。Debug 不是催 AI 多试几次，而是让每一轮都更接近真实问题。

### 第五步：交付前做一次 AI Code Review

快交付时，最危险的想法通常是：“主流程都通了，应该差不多了吧。” 这时候最值钱的，不是再让 AI 发挥一轮，而是让它做一次**轻量但聚焦的 Code Review**。

重点只看四件事：有没有**过度设计**，有没有和主目标无关的**无关改动**，有没有主流程之外的**边界遗漏**，以及页面文案、字段、按钮和说明之间是否存在**表达不一致**。这四类问题最容易混进交付物里，而且都很影响最后的可信度。

你可以直接这样提要求：请只从过度设计、无关改动、边界遗漏、表达一致性四个角度做简短审查；不要重写方案，不要扩展功能，只指出最影响交付质量的问题，并按优先级给出建议。这样它做的是交付前检查，不是新一轮需求膨胀。

前一章《[如何让 AI 帮你做 Code Review](../practical-skills/code-review-for-ai.md)》已经讲过具体提法；这里你只要记住一句话：**Review 的目的，不是显得专业，而是把“看起来能交”变成“真的敢交”。**

### 收尾：交付的不是一段代码，而是一套闭环

回头看这一章，我们真正走完的不是一个小页面，而是一整套流程：**需求描述 → 最小 PRD → 任务拆解 → AI 实现 → 调试修复 → Code Review → 交付收口**。

这才是案例真正想教你的能力。页面只是载体，闭环才是可迁移的方法。以后你做别的小需求，题目可以换，工具可以换，但只要还能把这条链路走完整，项目就不容易在“差不多行了”和“再改一点点”之间来回漂。

下一章，我们就继续处理闭环里最容易失控的那件事：Context 变长以后，AI 为什么开始跑偏。

## 实践

选一个你手边真实但不大的小需求，按本章闭环走一遍，而且每一步都只写最小版本：先写 5-7 行最小 PRD，再拆成 3-5 个任务，再给 AI 一段只做主流程的实现指令；如果第一版出问题，就补一条包含“现象、范围、约束、期望结果”的 Debug 反馈；最后再写一句交付前 Review 请求，只检查过度设计、无关改动、边界遗漏和表达不一致。

练习重点不是把题目做大，而是把闭环走顺。做完后问自己两个问题：每一步是不是已经具体到 AI 能直接行动？如果把题目换成另一个小需求，这套流程还能不能照搬？如果都差不多能答“能”，你练到的就是方法，不只是这一个案例。

## 小结

第四章的意义，不是给你一个现成答案，而是把前面讲过的目标收敛、任务设计、Debug 和 Review 串成一条完整工作流。这次案例已经从需求描述一路走到交付收口，你应该建立一个更稳的直觉：和 AI 协作时，结果质量往往不取决于某句神奇 Prompt，而取决于你能不能把过程组织成闭环。

当你开始这样工作，AI 才更像可调度的执行引擎，而你负责目标、边界和节奏。接下来进入第五章《[上下文一长，为什么 AI 就开始跑偏](../best-practices/context-limits.md)》，我们继续处理闭环里最常见、也最容易把项目带偏的问题。
